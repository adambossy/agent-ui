earendil-works/pi

A single tagged-union AgentEvent drives every surface — TUI, JSONL stdout, RPC mode, web UI library, and static HTML export. Every message update carries a full AssistantMessage snapshot, so consumers re-paint from the snapshot and don't need to track per-token diffs.

Correction. Pi does have a web UI — published as the npm package @earendil-works/pi-web-ui (formerly @mariozechner/pi-web-ui, now deprecated). It's built with mini-lit web components + Tailwind v4 and ships a Vite-driven example. The source lives in a separate package not currently in github.com/earendil-works/pi's public packages/ tree, but the example was installed from npm and run locally — screenshot below.

Web UI — source deep dive

Source: node_modules/@earendil-works/pi-web-ui/src/ (the unminified TypeScript inside the npm tarball — the package is shipped npm-only; no public git repo on the parent monorepo's main branch).

Distribution + stack — Lit web components, no React.
  • Package: @earendil-works/pi-web-ui v0.75.3 (271 versions of the deprecated @mariozechner/pi-web-ui precede it).
  • Stack: @mariozechner/mini-lit web components + Tailwind v4 + Lit 3 + lucide icons. No React, no Vue, no SvelteKit.
  • Tools shipped with the example: JavaScript REPL (sandboxed), document extraction (PDF/DOCX/XLSX/PPTX), artifacts (HTML/SVG/Markdown previews).
  • Example workflow: npm pack @earendil-works/pi-web-ui → extract → copy package/example → swap workspace deps for npm-published — runs at localhost:5173 via Vite.
Two-track rendering: stable list + RAF-batched streaming overlay — the defining architectural choice.

AgentInterface (components/AgentInterface.ts:19-403) hosts two separate components:

  • <message-list> renders completed messages, keyed by index (MessageList.ts:11-93). It re-renders only when a new message is appended.
  • <streaming-message-container> renders the single in-flight assistant message (StreamingMessageContainer.ts:6-104). Updates are batched to one requestAnimationFrame tick and the snapshot is deep-cloned via JSON.parse(JSON.stringify(...)) on each tick to defeat Lit's identity-based dirty check on nested arrays.

Every message_update from pi-agent-core carries a full AssistantMessage snapshot — same protocol as the CLI. Stable history + isolated overlay means partial updates only repaint the in-flight bubble, not the whole transcript.

Single-hook tool renderer contract (vs CLI's two-hook) — render(params, result, isStreaming).

The CLI exposes renderCall + renderResult as two hooks. The web UI fuses them into one:

// package/src/tools/types.ts:9-15
export interface ToolRenderer {
  render(
    params: unknown,
    result: ToolResult | undefined,
    isStreaming: boolean
  ): ToolRenderResult;
}

Branching on argument presence and isStreaming drives a four-state visual (streaming-args → executing → success → error). Per-tool renderers: BashRenderer.ts, DefaultRenderer.ts, CalculateRenderer.ts, GetCurrentTimeRenderer.ts, plus auto-registering javascript-repl.ts, extract-document.ts, and the artifacts renderer (isCustom: true).

Direct browser → LLM with surgical CORS proxying — most providers go direct; only three need a proxy.

AgentInterface.setupSessionSubscription (AgentInterface.ts:138-151) installs createStreamFn from utils/proxy-utils.ts:127-139. shouldUseProxyForProvider (proxy-utils.ts:19-51) only proxies:

  • Z-AI
  • OAuth Anthropic tokens (sk-ant-oat-*)
  • OpenAI-Codex

Everything else — OpenAI, Google, Groq, Ollama, LM Studio, etc. — goes direct from the browser to the provider. Custom local providers are autodiscovered (utils/model-discovery.ts:11-277): LM Studio over WebSocket via @lmstudio/sdk, others via /v1/models.

No central store, mini-lit in light DOM — Tailwind classes propagate freely.

All components opt out of shadow DOM via createRenderRoot() { return this } — repeated across Messages.ts:46, MessageList.ts:18, AgentInterface.ts:64, etc. Lets Tailwind classes propagate through the component tree without per-shadow-root style injection.

Reactive primitive: LitElement.requestUpdate() + property decorators. Persistence: async accessor Store classes (storage/store.ts:7-33) over an IndexedDBStorageBackend (storage/backends/indexeddb-storage-backend.ts:7-193) that splits sessions into sessions + sessions-metadata for fast listing.

Reasoning rendering — simpler than CLI / opencode — one collapsed block, shimmer while streaming.

package/src/components/ThinkingBlock.ts:6-43 is a single chevron-toggled block:

  • Always starts collapsed.
  • No keyboard shortcut.
  • No title extraction.
  • While streaming, the "Thinking..." label gets a 2-second Tailwind shimmer: app.css:47-58 defines the keyframe and applies bg-clip-text text-transparent animate-shimmer.

Animations are CSS-only — no JS-driven RAF ticks for spinners (just animate-spin for the tool spinner, animate-pulse for the streaming caret, the shimmer for thinking).

1. Top three findings

  1. Unified event protocol drives all surfaces. AgentEvent (packages/agent/src/types.ts:403-418) is a tagged-union stream of agent_start | turn_start | message_{start,update,end} | tool_execution_{start,update,end} | turn_end | agent_end. The TUI, the JSONL mode, the RPC mode, and the HTML exporter all consume the same events. Every message_update carries a full partial: AssistantMessage snapshot, so consumers always re-paint from the snapshot — no token-level diff tracking required.
  2. Tools are first-class UI citizens with two render hooks per tool. Every tool definition exposes renderCall(args, theme, ctx) and renderResult(result, options, theme, ctx) (see packages/coding-agent/src/core/tools/bash.ts:405-439, edit.ts:367-430). The shared ToolExecutionComponent (packages/coding-agent/src/modes/interactive/components/tool-execution.ts) keeps a per-tool state bag between invocations so renderers can mutate the previous frame's component tree instead of allocating new ones — that's why streaming bash output and growing edit diffs don't flicker. Background color encodes status (toolPendingBg | toolSuccessBg | toolErrorBg, tool-execution.ts:228-234). The edit tool kicks off an async on-disk diff preview the moment streamed args complete (edit.ts:381-390) so the user sees the would-be change before execution even starts.
  3. Reasoning has a dedicated content type that ships alongside text and tool calls. ThinkingContent (packages/ai/src/types.ts:230-238) appears in the same AssistantMessage.content array as text / toolCall blocks. The UI renders it italic in a thinkingText color via AssistantMessageComponent.updateContent (assistant-message.ts:73-122); Ctrl+T collapses it to a single "Thinking..." label that's customizable per-extension. Spacing between thinking and the next block is conditional — no blank row before tool boxes, blank row before text — preventing visual stutter (assistant-message.ts:97-119).

2. AgentEvent protocol

packages/agent/src/types.ts:403-418:

VariantMeaning
agent_startAgent harness booted.
turn_startNew user → assistant turn beginning.
message_startAssistant message slot opened.
message_update{ partial: AssistantMessage } — full snapshot per change.
message_endAssistant message finalised.
tool_execution_start{ toolCallId, name, args }
tool_execution_update{ toolCallId, partial }
tool_execution_end{ toolCallId, result | error }
turn_endTurn complete: status: completed | aborted | error
agent_endHarness shutting down.
Architectural choice: snapshots over deltas. Every message_update contains the complete current AssistantMessage. Renderers compare the new snapshot to the rendered tree and reconcile — no per-token append logic, no envelope state to maintain in the client.

3. Content types within an AssistantMessage

packages/ai/src/types.ts:

  • TextContent — { type: "text", text }
  • ThinkingContent (line 230-238) — { type: "thinking", text, signature? }
  • ToolCallContent — { type: "tool_call", id, name, input }
  • ImageContent, RedactedThinkingContent

The UI iterates message.content[] and emits one renderer per block. The same array carries reasoning, text, and tool calls — order is preserved exactly as emitted by the model.

4. Tool rendering — two hooks per tool

Every tool definition exposes:

renderCall(args, theme, ctx)       // "in flight" rendering
renderResult(result, options, theme, ctx)  // post-execution rendering

Examples: packages/coding-agent/src/core/tools/bash.ts:405-439, edit.ts:367-430.

The shared ToolExecutionComponent (packages/coding-agent/src/modes/interactive/components/tool-execution.ts) keeps a per-tool state bag between invocations so renderers can mutate the previous frame's component tree instead of allocating new ones. Streaming bash output and growing edit diffs don't flicker.

5. Reasoning rendering

ThinkingContent (packages/ai/src/types.ts:230-238) appears in the same AssistantMessage.content array as text / toolCall blocks. The UI renders it italic in a thinkingText color via AssistantMessageComponent.updateContent (assistant-message.ts:73-122).

  • Ctrl+T collapses to a single "Thinking..." label.
  • Collapsed label is customizable per-extension.
  • Spacing between thinking and the next block is conditional — no blank row before tool boxes; blank row before text — preventing visual stutter (assistant-message.ts:97-119).

6. State layers

No central store. State lives in three layers:

LayerLocationRole
Canonical transcriptAgent._state (agent.ts:509-547)Source of truth for messages, parts, tool results.
Persistence + eventsAgentSession (agent-session.ts:451)Writes to JSONL, emits AgentEvents.
UI handlesInteractiveMode class fields (interactive-mode.ts:240-280)Shadows component refs for in-place mutation.

Re-renders are coalesced via TUI.requestRender() → process.nextTick → throttled setTimeout (packages/tui/src/tui.ts:495-541).

7. Animations / spinners

  • 80 ms braille spinner default (packages/tui/src/components/loader.ts:11-12).
  • The bash tool drives its own per-second elapsed-timer redraw via setInterval (bash.ts:417-419) — useful for long-running commands.
  • Tool-box background colour transitions encode state, not motion.

8. Sub-agents and parallelism

  • No sub-agents in core. README.md:476 calls them an explicit non-goal — use extensions, tmux, or third-party packages.
  • Parallel tool calls are supported (agent-loop.ts:451), emit tool_execution_end in completion order.
  • The UI keys pending tools by toolCallId in a Map (interactive-mode.ts:280) so concurrent tool boxes update independently.

9. Conversation history

Conversation history is a JSONL tree (id + parentId per entry) supporting in-place branching, forking, cloning, and lossy compaction — see packages/coding-agent/README.md:230-272 for the user-facing model and packages/coding-agent/src/core/session-manager.ts for the file format.

10. Static HTML export — the only browser-renderable artefact

Template: packages/coding-agent/src/core/export-html/template.html (self-contained, no network).

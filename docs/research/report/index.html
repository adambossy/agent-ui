How agent UIs present a turn

Side-by-side look at six surfaces: three full agent UIs (ChatGPT, sst/opencode, earendil-works/pi — TUI and web), one closed-iframe widget (OpenAI ChatKit), and two reusable libraries (Vercel Chat SDK, Vercel AI Elements). Focused on what users actually see between hitting Enter and reading the answer.

The lineup

Web · Closed source

ChatGPT

Tools are invisible: the agent loop hides every tool-role message and surfaces work only as inline citation pills (150 ms fade-in). JSON-Patch stream over SSE; older turns are unmounted from the DOM.

TUI · Open source

sst/opencode

Three "parts" types (text, reasoning, tool) drive every render. Tools in two visual weights — inline and bordered block — with per-tool icons and live status colors. SolidJS + OpenTUI repaints only the changed text node per delta.

Read →

Web + TUI · Open source

earendil-works/pi

Pi ships a Lit-based web UI and a TUI on a shared AgentEvent protocol. Every message_update ships a full snapshot — no diff tracking. Web UI uses a stable list + rAF-batched streaming overlay, with deep-cloned snapshots to defeat Lit identity checks.

Read →

Closed iframe · OpenAI-only

OpenAI ChatKit

The npm packages are types + a 6.5 KB React wrapper. The actual UI is a closed iframe served from cdn.platform.openai.com. Same renderer that powers ChatGPT.com. Not usable outside OpenAI's infrastructure.

Read →

Next.js template · Open source

Vercel Chat SDK

A Next.js reference template + a constellation of npm libraries (ai, @ai-sdk/react, streamdown, resumable-stream). Libraries are platform-neutral; the template has ~5 single-line Vercel touchpoints. Killer feature: side-panel "artifacts".

React registry · Open source

Vercel AI Elements

A shadcn-style copy-paste registry of 47 React components for AI UIs. CLI shells out to shadcn add; source lands in your repo. Zero Vercel-infrastructure coupling. Includes Conversation, Message, Reasoning, Tool, PromptInput, voice, workflow canvas.

What stands out across all six

1. Three families: full UIs, libraries, and a closed iframe. — pick which kind of integration you want before picking a vendor.
  • Full agent UIs (ChatGPT, opencode, pi) — you use them as products or read their source to learn patterns. The mental model is "study the implementation."
  • Libraries / templates (Vercel Chat SDK, AI Elements, pi-web-ui as a library) — you compose your own UI from primitives. The mental model is "assemble pieces."
  • Closed iframe widgets (ChatKit) — you embed a vendor-controlled UI. The mental model is "rent the renderer." OpenAI's design here means ChatKit integrations are ChatGPT.com under a different config profile.
2. Three different answers to "what is a message?" — and that choice flows through everything.
  • ChatGPT treats a message as a single envelope with a content-string that grows via JSON-Patch append ops on /message/content/parts/0. Tools are separate hidden messages chained via parent_id.
  • opencode treats a message as an ordered array of typed Parts (text, reasoning, tool, plus 9 less-visible ones). Each part has its own lifecycle and renderer.
  • pi treats a message as an array of Content blocks (text, thinking, tool_call) all carried in the same AssistantMessage.content. Updates are full snapshots, not patches.
  • Vercel AI SDK (Chat SDK + Elements) standardises on UIMessage<Meta, Data, Tools> with a typed parts: UIMessagePart[] array. The part union covers text, reasoning, tool-<name> (with a 7-state machine), dynamic-tool, file, source-url, source-document, step-start/finish, and custom data-<key> channels.
  • ChatKit uses "thread items" with types assistant_message, user_message, client_tool_call, task (reasoning), widget, workflow, image_generation, generated_image, structured_input, end_of_turn. Server-authored, client-rendered widgets are the distinctive piece.
3. Reasoning rendering ranges from absent to elaborate. — and is highly user-controllable on the open-source ones.
  • ChatGPT free tier never shows a reasoning summary. server_ste_metadata reports did_auto_switch_to_reasoning: false.
  • opencode uses a three-state <Switch>: streaming shows "Thinking: <title>"; settled collapses to "+ Thought: <title> · 4.3s"; expanded shows full markdown.
  • pi (TUI) renders ThinkingContent inline italic; Ctrl+T collapses.
  • pi (web) shows one collapsed block; while streaming, a CSS keyframe shimmer animates the "Thinking..." label (no JS tick).
  • Vercel AI Elements ships <Reasoning> — Radix Collapsible that auto-opens on stream, auto-closes 1 s after, tracks duration, default trigger is "Thinking..." → "Thought for N seconds".
  • ChatKit emits thread.item.task for reasoning; rendering is inside the closed iframe.

5. Streaming protocols cluster around the same idea. — SSE everywhere; envelope shape varies.
  • ChatGPT: SSE event: delta records carry JSON-Patch ops (add, replace, append, patch) addressing a JSON-Pointer path inside the message envelope.
  • opencode: Two complementary event types — message.part.updated (full snapshot, persisted) and message.part.delta (transport optimisation, not persisted).
  • pi: Single message_update event with a full partial: AssistantMessage snapshot per change.
  • Vercel AI SDK: The "UI Message Stream protocol v1" — typed events (text-{start,delta,end}, reasoning-{start,delta,end}, tool-input-{start,delta,available}, tool-output-available, data-<key>) over SSE. Marker header x-vercel-ai-ui-message-stream: v1.
  • ChatKit: Custom thread-event protocol (ThreadItemDoneEvent, ThreadItemReplacedEvent, ProgressUpdateEvent, ClientEffectEvent, WidgetRootUpdated) — layered on the Responses API server-side.
6. Where the "renting vs owning" line falls. — it's about who controls what users see.
  • You own the source. opencode, pi, Vercel AI Elements (copy-paste), Vercel Chat SDK (template fork). You can patch a bug, customize a render path, host an air-gapped deployment.
  • You compose libraries. The AI SDK (ai + @ai-sdk/react) and pi-web-ui — runtime npm imports you can wrap.
  • You rent the renderer. ChatKit. You configure options, embed an iframe, and OpenAI controls every pixel. They ship updates without you redeploying — and they choose what features exist.
7. Vendor coupling — Vercel "AI SDK" stack is the least coupled. — ChatKit is the most.
  • AI Elements / AI SDK — zero Vercel-infrastructure dependencies. Apache-2.0. Works in Vite + any backend.
  • Chat SDK template — Next-locked but only ~5 single-line Vercel touchpoints (Blob, geo, BotID, OTel, Analytics) — all easily excised.
  • pi-web-ui — provider-agnostic (OpenAI, Anthropic, Google, Groq, Ollama, LM Studio…). Surgical CORS proxy only for Z-AI / OAuth-Anthropic / Codex.
  • ChatKit — four hard couplings to OpenAI: cdn.platform.openai.com iframe; platform.openai.com domain allowlist; api.openai.com/v1/chatkit/sessions for managed; Responses-API + Agents-SDK types in every example.
8. Subagents and parallelism — only opencode surfaces them visually. — pi rejects them; ChatGPT/ChatKit hide them.
  • opencode spawns child sessions via the task tool. The parent shows a live "↳ Reading /src/foo.ts" sub-line and a SubagentFooter with Parent / Prev / Next.
  • pi explicitly calls subagents a non-goal (CLI README). Parallel tool calls keyed by toolCallId in a Map.
  • Vercel Chat SDK exposes step-start / step-finish parts to mark multi-step agent boundaries but doesn't ship a UI for visualising sub-agent trees.
  • ChatKit uses thread.item.workflow for multi-step plans; rendering is opaque (inside the iframe).
9. Side-panel artefacts are catching on. — a workspace next to chat.
  • Vercel Chat SDK ships four built-in artifact kinds (text, code, image, sheet) with ProseMirror / CodeMirror / react-data-grid editors, version history, and a 60%-width side pane. Extension point is the Artifact<Kind, Metadata> class.
  • Vercel AI Elements exposes an Artifact component (Claude/v0-style container) — you bring your own data plumbing.
  • pi-web-ui has an ArtifactsPanel (HTML / SVG / Markdown previews via a sandboxed iframe).
  • ChatKit's widget DSL is the same idea expressed differently — server-authored JSON, client-rendered cards / lists / forms / tables, but locked to ChatKit's renderer.

Methodology & gaps

How each surface was studied — scope and tooling.
ChatGPT
Logged-in free-tier session driven via Playwriter MCP. 4 prompts sent; SSE bodies captured; DOM polled at 100 ms / 450 ms / 1.2 s / final.
sst/opencode
Cloned and read; no TUI was launched (would require an LLM key). Findings tied to file:line citations.
earendil-works/pi
Two paths captured: (a) pi --export rendered to static HTML and driven in Playwriter; (b) @earendil-works/pi-web-ui npm tarball example installed and run on Vite at localhost:5173. Source for both read in full.
OpenAI ChatKit
Three repos cloned + npm package extracted + CDN loader fetched and decoded. Starter app installs cleanly but requires an OpenAI key to run.
Vercel Chat SDK
vercel/chatbot template cloned (v3.1.0) + docs walked. Not deployed.
Vercel AI Elements
Monorepo cloned + per-component source read + 47 registry endpoints surveyed.
What's missing — what would unblock it.
  • ChatGPT's reasoning UI (Thinking / o-series): would need a Plus or Pro account.
  • opencode's animations observed live: would need to launch the TUI with an LLM key.
  • pi web-ui's live streaming (assistant turn finish was unreliable in Playwriter capture): the source already documents the rAF-batched / deep-clone strategy in detail.
  • ChatKit's actual rendered DOM (closed iframe): would need to embed and live-capture; the protocol surface is fully documented.

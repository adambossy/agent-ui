Cross-cutting concerns

The decisions that span the design — conflict policy, schema versioning, auth, and the explicit non-goals.

Conflict policy

Per-op declaration on the manifest:

PolicyWhen the server appliesUse for
reject-on-conflict (default)Only if baseSeq matches.Edits that must see fresh state (rename, replace).
commute-by-idIf the intervening ops touched different ids than this one.Add / remove / toggle on a keyed collection.
commute-by-deltaAlways; the intervening delta is added in.Counters, scores, anything additive.
lwwAlways; overwrites whatever's there.Direct sets where staleness is fine.

The default is reject-on-conflict — a safe failure mode. Extensions opt into laxer policies per op when they know the operation commutes. The wire envelope already carries baseSeq + opId — the same substrate a Yjs/Automerge layer would need — so a CRDT swap later doesn't break extensions that picked sensible policies up front.

Schema versioning

Three policies, in order of preference:

  1. Forward-compatible additions — new optional fields on the payload, new op variants. No version bump needed; old renderers ignore unknown fields and forward-emit unknown ops.
  2. In-place migrate — when an extension can lossless-migrate an older doc into the new shape, it sets manifest.schemaVersions = [1, 2] and provides migrate(d). The host calls it once per doc on first read.
  3. Distinct kind — for breaking changes that can't be auto-migrated, ship todo-list@2 as a new kind entirely. Both coexist; the user keeps using their old todo lists with the v1 manifest installed and creates new lists with v2.

The host renders an "unsupported version" fallback if a doc's schema version doesn't match any installed manifest's schemaVersions. Never crashes.

Auth scoping

Default: a doc is bound to the chat session that created it. Same trust boundary as the chat — anyone with the session id can read and write the doc. Authentication of the session is whatever the host has wired (cookie, bearer); the live-components subsystem doesn't add a new auth layer.

Optional opt-in: manifest.scope: "account" binds the doc to the authenticated account across sessions. The server then enforces ACLs on both POST and SSE subscribe. Not in MVP; the manifest field is reserved for forward compatibility.

Multi-tab: two tabs of the same session share the same SSE-stream broadcast. Two tabs in different sessions don't see each other's docs.

Explicit non-goals (and the forward seams left for each)

Non-goalForward seam left in the design
CRDTs (Yjs, Automerge)The wire envelope carries opId + baseSeq. Swapping in a CRDT means replacing the server-side reducer and the policy switch, not the envelope or the renderer contract.
Undo / redo as a first-class featureThe op log is the substrate. Extensions can build undo themselves by reversing op variants. A registry-level undo would mean a new manifest field (invert(op)) — additive.
Cross-extension composition (a todo inside a spreadsheet cell)One manifest per kind in the registry. Composition needs a separate "nested live component" primitive — different concern, not blocked by this design.
Real-time multi-user collabThe default scope: "session" rules it out; scope: "account" with an OCC reducer would let two users edit but conflicts would reject. Multi-user merge needs the CRDT swap above.
Runtime / lazy-loaded extensions on the web tierThe registry is a Map populated at import time. A runtime-loader could populate it from dynamic imports without changing the registry's interface — but build-time imports remain the recommended path for type safety.

None of these are blockers for the listed extensions (todo, counter, plus the hypothetical kanban / calendar / recipe / spreadsheet). They're known limits, surfaced so future contributors don't try to solve them inside this design.

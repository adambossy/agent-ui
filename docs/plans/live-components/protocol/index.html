Live components  /  Protocol & dispatch

Branch · Protocol

Wire envelope & dispatch path

One new data-live-op channel on the existing AI SDK SSE stream. One new POST /api/live/:kind/:docId/op endpoint for user-originated ops. Same OCC discipline as a CRDT, simpler semantics for MVP.

Requirements

  1. A backend implementer can write the live-op handler using only this page.
  2. Conflicting writes (LLM and user mutating the same doc in the same window) are detected and surfaced, never silently merged into something neither party intended.
  3. Two tabs of the same chat see ops in the same order and converge on identical state.

The wire envelope

One new data-* channel, comparable to data-subagent-event:

type LiveOpEvent =
  | { kind: "doc-init";
      docId: string; componentKind: string;
      schemaVersion: number; payload: unknown }
  | { kind: "op-applied";
      docId: string; opId: string; op: unknown;
      serverSeq: number; origin: "llm" | "user" }
  | { kind: "op-rejected";
      docId: string; opId: string; reason: string }
  | { kind: "doc-snapshot";
      docId: string; schemaVersion: number;
      payload: unknown; serverSeq: number };

// Sent as:
//   data: {"type":"data-live-op","data":LiveOpEvent,"transient":false}
  • doc-init — opens a doc on this stream. Sent once per (stream, docId).
  • op-applied — confirmed op, in serverSeq order. origin tells the client whether to drop a matching pendingOp (user) or just apply (llm).
  • op-rejected — rolls back the matching optimistic op. Carries a short reason.
  • doc-snapshot — catch-up record after resume or sequence skip. Replaces local state wholesale.

The marker header x-vercel-ai-ui-message-stream: v1 and SSE plumbing are unchanged. Cross-link: live-components-design.md §4.

LLM emit path

sequenceDiagram participant L as LLM participant T as Tool execute() participant S as Server participant C as Client L->>T: call todo.addItem(text="...") T-->>S: returns { ops: [{ kind: "add-item", ... }] } S->>S: validate via opCodec; apply via reducer; persist S-->>C: data-live-op { kind: "op-applied", origin: "llm", serverSeq: N+1 } S-->>L: tool-output-available C->>C: reducer.apply(op) → renderer rerenders

The tool's execute returns { ops: Op[] } — never new state. The server is the only writer to the doc. The tool's tool-output-available event still flows back to the LLM, but its content is whatever the server chooses (typically a confirmation like { docId, applied: N }); the doc's state reaches the LLM via the persistence layer's "doc latest" query that the agent's system prompt reads, not via tool output.

User emit path

sequenceDiagram participant U as User participant R as Renderer participant ST as Doc store participant N as LiveOpClient participant S as Server U->>R: click toggle R->>ST: dispatch(op) ST->>ST: optimistic apply + push to pendingOps ST-->>R: rerender immediately R->>N: send(op, baseSeq, opId) N->>S: POST /api/live/todo-list/:docId/op S->>S: validate; OCC on baseSeq; apply; persist alt accepted S-->>N: 200 { serverSeq: N+1 } S-->>ST: data-live-op { kind: "op-applied", origin: "user", opId } ST->>ST: drop pending entry; advance serverSeq else rejected S-->>N: 409 { reason } S-->>ST: data-live-op { kind: "op-rejected", opId, reason } ST->>ST: rollback to last confirmed state ST-->>R: rerender with prior state + error end

POST body:

POST /api/live/:kind/:docId/op
Content-Type: application/json

{
  "opId": "<uuid>",
  "op": { "kind": "toggle-item", "id": "..." },
  "baseSeq": 42,
  "sessionId": "<chat session id>"
}

The 200 response is informational; the authoritative confirmation arrives via the SSE op-applied broadcast, which all subscribed tabs receive simultaneously.

Conflict handling

Optimistic concurrency on baseSeq. If the LLM applied an op between the user reading state and POSTing, the server's stored serverSeq is higher than the user's baseSeq. The server then consults the manifest's per-op conflict policy:

PolicyWhat the server does on stale baseSeq
commute-by-idApply the op anyway if it touches a different entity than the intervening ops did. E.g. toggling item A and toggling item B commute.
reject-on-conflict (default)Reject with op-rejected; client rolls back. Safe default — the renderer can surface a "list changed under you" hint.
lwwApply unconditionally, last-write-wins. Useful for non-destructive ops like cursor position.

Policy is declared per op variant by the extension in the manifest. Default is reject-on-conflict. CRDT-shaped — the wire envelope already carries opId + baseSeq, which is the substrate a real CRDT layer would need; swapping in Yjs later means replacing the server-side reducer without changing the envelope.

Multi-tab consistency

Two tabs of the same chat session both subscribe to the same SSE stream. Each receives the same op-applied records in the same serverSeq order; both reduce identically. Pending optimistic ops are per-tab — Tab A's pending-toggle is invisible to Tab B until the server confirms and broadcasts. The single source of truth (server's persisted doc) keeps them convergent.

Resume: a refresh causes a new SSE subscribe. The server detects the gap (last-seen serverSeq < current) and emits doc-snapshot to bring the tab up to date in one frame instead of replaying the op log. Cross-link: resumable-streams.md.

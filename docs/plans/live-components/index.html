Live components: schema-typed UI the LLM and user co-edit

Third parties ship a manifest — a JSON schema, an op union, the LLM tools that emit ops, and a renderer — and call registerLiveComponent(manifest) at module load. Both the LLM (via tool calls) and the user (via clicks in the rendered UI) mutate the same document through the same server-applied pipeline. The chat core never imports anything kind-specific. Subsequent LLM turns see the latest state because there is one source of truth.

Requirements

  1. A plugin author can ship a new component (todo list, kanban, calendar, …) by writing one TypeScript module and adding one import line in the host's src/extensions/index.ts.
  2. The chat application never contains any code specific to a particular component. Removing an extension removes only its module and the import line; the rest of the chat keeps working.
  3. When the LLM updates a document via a tool, the user sees the change in the rendered component within one network round-trip.
  4. When the user toggles a checkbox or edits an item in the rendered component, the LLM's next turn sees the updated state without any extra plumbing.
  5. Two different components can coexist in the same chat — the registry dispatches by kind without leaking either extension's concerns into the other.

The thing this enables

A user asks Claude to "make me a packing list for Tokyo." The chat surface renders an interactive todo list inline; Claude streams items in via todo.addItem tool calls; the user checks off items they already packed; tomorrow they ask Claude "what's left?" and Claude sees the updated list. Same plumbing supports a kanban board, a calendar, a recipe builder, a counter, a spreadsheet — anything whose state fits in a typed JSON document mutated by discrete operations.

This is structurally a third pattern, distinct from artifacts (stream-of-deltas, versioned, read-only) and from generative UI (one-shot per tool call, no persistence). It's a shared mutable document with two writers — the LLM and the user — both routed through a single server-applied op pipeline.

How it fits with what's already here

The closest existing precedent in this codebase is the subagent extension (app/src/state/subagentStore.ts + app/src/components/tools/SubagentTool.tsx). That system already demuxes a typed channel (data-subagent-event) off the SSE stream into a Zustand store keyed by parentToolCallId, with a custom renderer that subscribes selectively. Live components follow the same shape and add the missing piece: an outbound dispatch path so user clicks issue ops that travel back to the server.

  • One new SSE data channel: data-live-op.
  • One new Zustand store: liveDocStore, keyed by (kind, docId).
  • One new generic renderer: LiveComponentHost, registered for every kind via the existing registerToolRenderer.
  • One new HTTP endpoint: POST /api/live/:kind/:docId/op.

How to read this plan

Start with Architecture for the data flow, then Protocol & dispatch for the wire format. Registration + Plugin author guide answer "what does a plugin builder do." The two Sample extensions are end-to-end worked examples. Mock backend is the dev-time server. Cross-cutting covers schema versioning, conflict policy, and explicit non-goals. Phasing sequences the work. Assumptions records what we decided vs. defaulted vs. left open.

Out of scope

  • CRDTs. Last-write-wins via server-authoritative ops, with optimistic-concurrency on a baseSeq. Forward-compatible with Yjs/Automerge swap-in later; not in MVP.
  • Undo/redo as a first-class feature. The op log is the substrate; extensions can build their own.
  • Cross-extension composition. A todo renderer cannot embed a spreadsheet inside a cell. One manifest per kind; nested live components need a separate composition primitive later.
  • Real-time multi-user collab. Two humans editing the same doc concurrently is not supported. Two tabs of the same user is.
  • Runtime / lazy-loaded extensions. Build-time only for the web tier; the CLI tier (pi-style pi install) is where runtime loading belongs, and we're not building that.

In-memory mock backend

A Map<docId, { kind, schemaVersion, payload, serverSeq, subscribers }> in the Vite-plugin process. Handles POST /api/live/:kind/:docId/op for user ops, applies the matching manifest's mockServerReducer, broadcasts op-applied on every subscribed SSE stream. Doc state is lost on dev-server restart.

Requirements

  1. A frontend developer can demonstrate live editing of a todo or counter end-to-end without a real backend.
  2. The mock behaves identically to the real backend on the wire — same op-applied / op-rejected events, same OCC semantics.
  3. Resetting state means restarting vite. No additional config or cleanup commands required.

In-memory data model

// app/src/mock/live/store.ts
type LiveDocRecord = {
  docId: string;
  kind: string;
  schemaVersion: number;
  payload: unknown;
  serverSeq: number;
  subscribers: Set<ServerResponse>;   // SSE responses to broadcast to
};

const docs = new Map<string, LiveDocRecord>();

The mock's Map sits in the Vite plugin's Node process. It survives Vite HMR (the plugin module is preserved) but not a full server restart. That's the intended ergonomic — restart = clean slate.

POST endpoint

// app/src/mock/vite-plugin.ts (additions)
if (method === "POST" && /^\/api\/live\/[^/]+\/[^/]+\/op$/.test(url)) {
  const [, , , kind, docId] = url.split("/");
  const body = await readJson(req) as {
    opId: string; op: unknown; baseSeq: number; sessionId: string;
  };

  const manifest = getLiveComponentForServer(kind);  // re-uses extension's mockServerReducer
  if (!manifest) return send(res, 404, { error: "unknown kind" });

  const doc = docs.get(docId);
  if (!doc) return send(res, 404, { error: "unknown doc" });

  if (body.baseSeq !== doc.serverSeq) {
    const policy = manifest.conflictPolicyFor(body.op);
    if (policy === "reject-on-conflict") {
      broadcast(doc, { kind: "op-rejected", docId, opId: body.opId, reason: "stale baseSeq" });
      return send(res, 409, { reason: "stale baseSeq" });
    }
    // otherwise fall through and let policy handle it
  }

  const op = manifest.opCodec.parse(body.op);   // throws → 400
  doc.payload = manifest.mockServerReducer(doc.payload, op);
  doc.serverSeq += 1;
  broadcast(doc, { kind: "op-applied", docId, opId: body.opId, op, serverSeq: doc.serverSeq, origin: "user" });
  return send(res, 200, { serverSeq: doc.serverSeq });
}

Broadcast over SSE

Each chat session's POST /api/chat opens an SSE response that stays open for the turn. While open, the SSE writer subscribes the response to any docs the LLM touches:

function broadcast(doc: LiveDocRecord, event: LiveOpEvent) {
  for (const res of doc.subscribers) {
    res.write(`data: ${JSON.stringify({
      type: "data-live-op", data: event, transient: false,
    })}\n\n`);
  }
}

After the chat turn ends ([DONE]), the SSE response closes; subscribers are pruned. A new turn opens a new SSE; if the same doc is referenced, it's re-subscribed. For "the user is mid-edit while a turn is not in flight" — same flow, the POST endpoint broadcasts to any still-open SSE that happens to be subscribed. In single-session use, that's the current chat's stream.

LLM tool path

The mock plugin's existing scripted-turns infrastructure (src/mock/turns/*.ts) gains a helper:

// app/src/mock/turns/lib/live.ts
export function applyLiveOp(
  res: ServerResponse,
  kind: string, docId: string, op: unknown
): { serverSeq: number; payload: unknown } {
  const doc = docs.get(docId) ?? createDoc(kind, docId);
  doc.payload = getManifestForServer(kind).mockServerReducer(doc.payload, op);
  doc.serverSeq += 1;
  doc.subscribers.add(res);
  ev(res, { type: "data-live-op", data: {
    kind: "op-applied", docId, opId: crypto.randomUUID(),
    op, serverSeq: doc.serverSeq, origin: "llm",
  }, transient: false });
  return { serverSeq: doc.serverSeq, payload: doc.payload };
}

A scripted "todo-list" turn calls applyLiveOp from inside the SSE streaming loop. The LLM-equivalent in the mock is just a hand-written script that uses the same broadcast helper the real POST endpoint uses — same wire output.

Restart behavior

  • Vite HMR — the mock plugin module is preserved; the docs Map survives. Edits to app/src/components/* rebuild the client but the doc store keeps state.
  • Full npm run dev restart — process exits; the docs Map is GC'd; next start is empty.
  • Browser refresh — the client's liveDocStore resets, then it reconnects the SSE. The server emits doc-snapshot for any doc referenced in the rehydrated chat history.

If a developer wants persistence, the upgrade path is documented but not implemented in MVP: drop in a better-sqlite3 backend at app/src/mock/live/sqlite-store.ts with the same interface. The Vite plugin code paths don't change.

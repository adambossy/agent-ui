todo-list — the rich sample

Mutation-heavy. Four ops, four LLM tools, interactive renderer with inline edit + checkbox + delete + add. Conflict policy is commute-by-id so two ops on different items never reject.

Requirements

  1. The user can ask Claude "make me a packing list for Tokyo" and watch items stream in.
  2. The user can check, uncheck, edit, and delete items in the rendered list at any time.
  3. The user can ask Claude "what's left?" and Claude correctly enumerates only the unchecked items.

Schema

const TodoItem = z.object({
  id: z.string().uuid(),
  text: z.string().min(1).max(500),
  done: z.boolean(),
  createdAt: z.string().datetime(),
});

const Todo = z.object({ items: z.array(TodoItem) });
type Todo = z.infer<typeof Todo>;

Ordering: items are kept in insertion order. Reordering is deferred to a follow-up op (reorder-item) — not in MVP.

Op union

OpPayloadConflict policy
add-item{ id, text }commute-by-id — adds with different ids commute
toggle-item{ id }commute-by-id — toggles on different items commute; two toggles on the same id reject
edit-item{ id, text }reject-on-conflict — concurrent edits on the same id need user attention
remove-item{ id }commute-by-id

LLM tools

One tool per op, plus a coarser todo.createList that emits a doc-init op for the LLM to start a fresh list:

{
  "todo.createList":  z.object({ title?: string }),
  "todo.addItem":     z.object({ docId: string, text: string }),
  "todo.toggleItem":  z.object({ docId: string, itemId: string }),
  "todo.editItem":    z.object({ docId: string, itemId: string, text: string }),
  "todo.removeItem":  z.object({ docId: string, itemId: string }),
}

Each tool's execute returns { docId, ops } — never new state. The server applies, persists, broadcasts. The tool-output-available event the LLM receives is a tiny confirmation envelope, not the new payload.

System prompt hint for the LLM (registered alongside the tools): "Use todo.createList first; subsequent items go through todo.addItem against the returned docId. Batch related items into a single tool call when possible to reduce round-trips."

Renderer behaviour

  • Checkbox — dispatches { kind: "toggle-item", id } on change. Optimistic apply, line-through immediately, server confirms within ~200ms.
  • Inline text edit — contentEditable or <input> blurs to dispatch edit-item. Local debounce 400ms.
  • Delete button — small "×" on hover (desktop); permanently visible on touch (mobile). Single click dispatches remove-item with no confirmation — the op log + server is the undo path, not a modal.
  • Add row — a permanent empty row at the bottom; Enter dispatches add-item and clears the input.
  • Pending indicator — items with a matching opId in meta.pending get a subtle opacity-70 until confirmed.
  • Error surface — if meta.lastError is set, a one-line banner above the list shows it for 3s then clears.

Demo flow (mock backend)

  1. User: "Make a packing list for a 3-day Tokyo trip in October."
  2. LLM calls todo.createList({ title: "Tokyo trip" }); server emits doc-init; renderer appears in the message.
  3. LLM calls todo.addItem ~10 times, batched across 2 model steps. Each broadcasts op-applied; renderer shows items appearing one by one.
  4. User checks "passport" and edits "comfortable shoes" → "running shoes." Two op-applied records broadcast with origin: "user".
  5. User: "What's left to pack?" LLM reads the persisted doc (via the agent's system-prompt context), enumerates unchecked items.
  6. Restart the dev server → doc is lost (in-memory mock). Same prompt restarts from empty.

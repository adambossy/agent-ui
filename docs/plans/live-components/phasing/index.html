Phasing: MVP → iterations

Four phases. MVP gets both samples working against the mock backend. Phase 2 adds the second sample as proof of multi-kind. Phase 3 wires the real backend. Phase 4 is the production-concerns lap.

Requirements

  1. Each phase ends with a runnable, demonstrable app.
  2. No phase requires the next to be valuable. We can stop after any phase and the work shipped so far is still useful.

MVP (Phase 1) — ~3 days

Goal: todo-list end-to-end against the mock backend. Subagent extension keeps working; counter not yet shipped.

  • app/src/live/registry.ts, store.ts, host.tsx, client.ts, types.ts.
  • app/src/extensions/index.ts populated with one entry (todo-list).
  • app/src/mock/vite-plugin.ts extended with POST /api/live/:kind/:docId/op.
  • app/src/mock/turns/todo-list.ts — a scripted turn that creates a todo doc and streams ~10 items via todo.addItem.
  • Empty-state demo button on /: "Make me a packing list for Tokyo."

Exit criteria. Click the demo button. Watch items stream in. Toggle one, edit one, delete one, add one. Refresh — doc rehydrates from the mock's in-memory store (still present since dev server hasn't restarted). Restart dev server — doc is gone, click button again, clean slate.

Phase 2: second extension — ~1 day

Goal: prove multi-kind dispatch. Ship the counter extension; add a demo turn that creates one of each in the same chat.

  • app/src/extensions/counter/* — schema, reducer, tools, renderer, manifest.
  • app/src/extensions/index.ts grows by one import line.
  • app/src/mock/turns/multi-doc.ts — a scripted turn that creates a todo + a counter and updates each.

Exit criteria. One chat session contains one todo list, one counter, and a subagent expansion side-by-side. Edits to each affect only that doc.

Phase 3: real backend — ~2 days

Goal: live components work against the agent_harness (or vercel/chatbot template, for now) backend. Requires backend-side work that's out of scope for the frontend plan; the frontend side of this phase is a small adapter.

  • Detect VITE_BACKEND=real at the LiveOpClient level; redirect POSTs to the configured backend URL.
  • Backend implements POST /api/live/:kind/:docId/op, validates against the extension's published JSON Schema, applies via its own reducer, broadcasts on SSE.
  • Persistence: backend stores (docId, kind, schemaVersion, payload, serverSeq) in whatever store it uses (Postgres, SQLite, etc).

Exit criteria. The same todo + counter demos work with VITE_BACKEND=real. Refreshing the page rehydrates the doc from the real backend; restarting the dev server does not lose state.

Later phases

  • Phase 4 — Tool approvals on live ops. When the LLM tries to delete an entire list, the user gets an approval prompt in the composer (matches the subagent approval pattern from the parent plan).
  • Phase 5 — Schema-version migrations. Author a todo-list@2, ship the migrate() function, watch v1 docs rehydrate as v2.
  • Phase 6 — Optional persistent mock. Drop in better-sqlite3 at app/src/mock/live/sqlite-store.ts with the same interface; toggle via env var.
  • Phase 7 — Account-scoped docs. Wire scope: "account" through the auth layer when one exists.
  • Phase 8 — CRDT swap. Replace the OCC reducer with a Yjs-backed one. Wire envelope unchanged.

Phases 4 onward are independent; pick whichever the user needs next.

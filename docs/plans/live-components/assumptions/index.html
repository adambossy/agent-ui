Remaining open questions

  1. How does the LLM "see" the current doc state in its next turn? The doc is server-persisted; the agent's prompt-builder needs to include the relevant doc(s) in context. Options: include the full payload as a system message; include a structured summary; let the LLM call a todo.read tool. Default for MVP: full payload as a system message scoped to recently-touched docs. Needs verification once the real backend lands.
  2. Pagination / size limits on payload. A pure-payload approach hits token-budget walls fast for big docs (a 200-item kanban). The forward seam is a manifest field summarize?(state): string that the agent's context-builder calls instead. Not in MVP.
  3. How does the tool know which docId to target? Three options: the LLM is told the docId via tool input schema (current sketch); the host injects it as a hidden tool arg; the server picks "the most recent doc of this kind." First option is most explicit; second is what most agent frameworks do; third is dangerous. Need to land on one before shipping more than the MVP demos.
  4. What happens when an extension throws inside its renderer? Currently undefined. Need an error boundary at the LiveComponentHost level so a broken renderer doesn't crash the whole chat. Easy fix in MVP.
  5. Is there a "discard this doc" affordance? The user might want to delete a doc the LLM created. Right now nothing in the design says how. A simple "×" on the renderer chrome (host-provided, not per-extension) plus a server-side doc-tombstone op variant solves it.
  6. Should the renderer have control over its own chrome? Pi has isCustom: true for full-bleed renderers; this design assumes a host wrapper by default. Should we lift the same flag onto the manifest (renderer.bare?: true)? Probably yes; small addition.

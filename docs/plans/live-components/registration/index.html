Live components  /  Registration

Branch · Registration

Build-time imports + global registry

Extension authors call registerLiveComponent(manifest) at module top level. The host adds one import line to src/extensions/index.ts per plugin. Type-safe end to end. No dynamic loading. Adopted directly from pi-web-ui's pattern; their CLI tier handles runtime-loaded extensions on a different surface.

Requirements

  1. Adding a new extension to a running app takes exactly one new import line in the host's extensions file.
  2. Removing an extension takes exactly one line deletion; the app keeps building and running.
  3. The host application's import graph is the authoritative answer to "which extensions are loaded right now?"

Pi's precedent (verbatim references)

Pi-web-ui's src/tools/renderer-registry.ts defines a single global Map:

export const toolRenderers = new Map<string, ToolRenderer>();

export function registerToolRenderer(toolName: string, renderer: ToolRenderer): void {
  toolRenderers.set(toolName, renderer);
}
export function getToolRenderer(toolName: string): ToolRenderer | undefined {
  return toolRenderers.get(toolName);
}

Their own built-ins self-register via side-effect imports (src/tools/index.ts):

import "./javascript-repl.js";   // Auto-registers the renderer
import "./extract-document.js";  // Auto-registers the renderer
registerToolRenderer("bash", new BashRenderer());

And consumer apps register their custom message types with an explicit call (example/src/main.ts line 31):

import { registerCustomMessageRenderers } from "./custom-messages.js";
registerCustomMessageRenderers();          // explicit kick before mount

We adopt the same shape, renamed for the live-component scope:

// app/src/live/registry.ts
export const liveComponents = new Map<string, LiveComponentManifest<any, any>>();

export function registerLiveComponent<S, Op>(m: LiveComponentManifest<S, Op>) {
  liveComponents.set(m.kind, m);
}
export function getLiveComponent(kind: string)
  : LiveComponentManifest<any, any> | undefined {
  return liveComponents.get(kind);
}

Peer-dep contract

An extension package's package.json:

{
  "name": "@my-org/live-todo-list",
  "version": "0.1.0",
  "type": "module",
  "exports": {
    ".": "./dist/index.js",
    "./tool": "./dist/tool.js"          // optional split; see below
  },
  "peerDependencies": {
    "@agent-ui/live-host": "^0.1.0",    // our package, when published
    "react":                "^19.0.0",
    "ai":                   "^6.0.0",
    "zod":                  "^3.0.0"
  },
  "sideEffects": ["./dist/index.js"]    // ← crucial: prevents tree-shaker
                                        //   from dropping the side-effect import
}

The sideEffects array is what tells Vite / Rollup / esbuild that the side-effect import in the consumer's extensions/index.ts matters. Without it, an over-aggressive tree-shaker could remove import "@my-org/live-todo-list" entirely because nothing references its exports.

Side-effect import vs explicit register call

Both shapes are supported. Extension authors choose:

PatternHost codeWhen to use it
Side-effectimport "@my-org/live-todo-list";Default. Smallest host code. Works if the package marks sideEffects.
Explicit callimport { register } from "@my-org/live-todo-list"; register();When ordering matters (e.g. one extension depends on another being registered first) or when the package wants registration to be lazy.
Tool factoryimport { createTodoTools } from "@my-org/live-todo-list/tool";For the tool half (LLM-callable). Tools are constructed per-agent because they may carry instance state.

Pi uses all three at the same time — side-effect for built-in renderers, explicit call for custom-message renderers, tool factory for the JavaScript REPL. We adopt the same split: renderer = global, tool = per-agent.

Where the registration lives in the host

// app/src/extensions/index.ts
//
// The host's extension manifest. Adding an extension is one
// import line here. Removing is one deletion.
//
import "@my-org/live-todo-list";        // side-effect: registers the renderer
import "@my-org/live-counter";          // ditto

// If any extension needs the tool factory at agent setup, re-export it
// from here so ChatScreen finds it in one place:
export { createTodoTools }    from "@my-org/live-todo-list/tool";
export { createCounterTools } from "@my-org/live-counter/tool";

That file is imported once from app/src/main.tsx before the React tree mounts:

// app/src/main.tsx
import "./extensions";          // populates the live-component registry
// … then createRoot(...).render(…)

The chat core (ChatScreen, Message, LiveComponentHost) reads from liveComponents.get(kind) at render time and from getLiveTools() at agent setup. It never imports any specific extension.

Architecture overview

Five pieces. A registry of manifests, a global doc store, a generic host renderer, a wire envelope, and a single-endpoint server. Adding an extension touches the registry; the rest is wired once and never changes.

Requirements

  1. A developer can sketch the data flow on a whiteboard from this page alone — no other reading required.
  2. The architecture composes with the existing subagent extension and the existing chat-screen plumbing; nothing here invents a new transport.

The five pieces

PieceWhat it isOwner
LiveComponentRegistryOne map from kind to LiveComponentManifest. Populated by extension modules at import time.Chat core, knows only the type — never a specific kind.
LiveComponentManifest<S, Op>Schema + op codec + tool factories + renderer + reducer.Extension author. See manifest.
liveDocStoreOne global Zustand store, keyed by (kind, docId). Reduces ops via the manifest's reducer.Chat core. See state store.
LiveComponentHostGeneric React renderer registered for every tool-${kind}. Resolves the manifest, subscribes to the doc, provides dispatch.Chat core.
LiveOpClientPOSTs user ops, applies them optimistically, reconciles with the server's broadcast.Chat core.

Data flow

flowchart LR llm[LLM] -->|tool call| server user[User click in renderer] -->|POST /api/live/.../op| server server[Agent server
applies ops, persists] -->|data-live-op| sse[SSE stream] sse --> reducer[liveDocStore reducer] reducer --> render[Renderer rerenders] user --> opt[Optimistic local apply] --> render server -. on conflict .-> rollback[op-rejected] --> reducer

One pipeline both writers go through. The LLM's execute() returns ops; the user's click POSTs ops. Server applies, persists, broadcasts. The same SSE stream carries the broadcast back. The next LLM turn sees the latest doc because the server is the only source of truth.

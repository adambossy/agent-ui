The manifest

Everything a third-party extension hands the chat app — schema, op union, op codec, LLM tools, renderer, and a pure reducer — bundled into one object passed to registerLiveComponent().

Requirements

  1. A plugin author can write a complete extension by filling in the fields of this one type. No other contracts need to be learned.
  2. The type is parameterised over the document shape and the op union so TypeScript catches mismatches between the reducer, the tools, and the renderer.

Shape

export type LiveComponentManifest<S, Op> = {
  /** Stable string identifier. Used as the AI SDK tool prefix
   *  (`tool-${kind}`) and as the doc-store key. */
  kind: string;                            // "todo-list" | "counter" | …

  /** Versions this manifest can render. */
  schemaVersions: number[];                // [1]; [1, 2] after migration

  /** Construct a fresh empty doc payload. */
  initialState: (seed?: unknown) => { schemaVersion: number; payload: S };

  /** Pure reducer. Same input → same output. No side effects. */
  reducer: (state: S, op: Op) => S;

  /** Runtime validation + parsing of an unknown op into a typed Op. */
  opCodec: OpCodec<Op>;

  /** AI SDK tools the LLM can call. Tools return `{ ops: Op[] }`,
   *  never new state directly. */
  tools: Record<string, LiveTool<Op>>;

  /** React component that displays the doc and dispatches ops on
   *  user interaction. */
  renderer: React.ComponentType<{
    doc: S;
    dispatch: (op: Op) => void;
    meta: LiveDocMeta;     // docId, kind, serverSeq, pending, status
  }>;

  /** Optional: migrate older payloads into the current schema. */
  migrate?: (d: { schemaVersion: number; payload: unknown })
    => { schemaVersion: number; payload: S };

  /** Optional: when running with mock backend, server-side reducer
   *  applied identically. Defaults to the client-side reducer. */
  mockServerReducer?: (state: S, op: Op) => S;
};

The op codec

Ops cross trust boundaries — they arrive from the SSE stream (unknown), from POST bodies (unknown), and from the LLM's tool output (unknown). The codec parses unknowns into typed Op at every boundary, throwing on shape mismatch.

export interface OpCodec<Op> {
  parse(input: unknown): Op;          // throws on invalid
  safeParse(input: unknown): { ok: true; op: Op } | { ok: false; error: string };
}

// Default Zod-backed implementation provided:
export function zodCodec<Op>(schema: z.ZodType<Op>): OpCodec<Op>;

Extension authors can use Zod (recommended), TypeBox, hand-rolled validators, or anything that implements the interface. The chat core never touches op contents — only routes parse failures back up via op-rejected.

Tools versus ops

The relationship is N-to-N, not 1-to-1. Common patterns:

ShapeExample
One tool → one optodo.toggleItem → { kind: "toggle-item", id }
One tool → batch of opstodo.importChecklist → [{ kind: "add-item" }, { kind: "add-item" }, …]
Op without a dedicated tool{ kind: "reorder" } emitted only by the user via drag-drop; no LLM tool

The contract that holds: tools never produce state, only ops. The reducer is the only function that turns ops into state, and it runs on the server. This is the load-bearing decision from the design proposal at notes/live-components-design.md.

Schema versioning

Manifests advertise which schema versions they handle. On doc-init the host looks at schemaVersion and either:

  1. Renders directly if the manifest declares that version in schemaVersions.
  2. Calls manifest.migrate() if defined and the version is older.
  3. Renders a "unsupported version" fallback in the host chrome — never blows up.

Hard schema bumps register a distinct kind: todo-list@1 and todo-list@2 are independent extensions. The server picks the right one based on the persisted schemaVersion.

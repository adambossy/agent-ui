Leaf · Architecture

The doc store

One global Zustand store keyed by (kind, docId). The chat core owns it; extensions never ship state infrastructure. Mirrors the pattern from the existing subagentStore.ts.

Requirements

  1. Updating one doc never re-renders the renderer of another doc.
  2. A renderer can subscribe to a slice of its doc (e.g. a single item's done state) and only re-render when that slice changes.
  3. Pending optimistic ops are visible to the renderer but distinguishable from confirmed state.

Shape

type DocStatus = "hydrating" | "ready" | "desynced" | "error";

type LiveDocEntry<S, Op> = {
  kind: string;
  docId: string;
  schemaVersion: number;
  payload: S;                      // the confirmed state from the server
  pendingOps: Array<{ opId: string; op: Op }>;   // optimistic local
  serverSeq: number;               // highest applied server sequence
  status: DocStatus;
  lastError?: string;
};

type LiveDocStoreState = {
  docs: Record<string, LiveDocEntry<unknown, unknown>>;
  apply(event: LiveOpEvent): void;
  enqueue(kind: string, docId: string, opId: string, op: unknown): void;
  rollback(docId: string, opId: string, reason: string): void;
};

Key format: `${kind}::${docId}`. The store doesn't know any specific schema; it stores unknown payloads and lets the manifest's reducer (looked up from the registry) decide how to apply ops.

Why (kind, docId) and not just docId

DocIds are opaque server-issued UUIDs. The kind is needed because:

  • Renderer dispatch — the host renders tool-${kind} parts; the kind is what resolves the manifest, which carries the reducer.
  • Validation — an op codec is per-kind; you can't validate an op without knowing which schema it targets.
  • Authorisation — future ACL work likely scopes by kind ("this user can edit todo lists but not workflow definitions").

Storing kind alongside the docId also means the store survives a missing manifest cleanly: the renderer can show "unknown component kind: foo" without needing the registry to know what foo is.

Selector hooks

// Whole doc — for rendering the top-level structure
const doc = useLiveDoc<Todo>(kind, docId);

// Single slice — for narrow re-render boundaries
const itemCount = useLiveDocSelector(
  kind, docId,
  (s: Todo) => s.items.length
);

// Doc metadata — pending status, server seq, errors
const meta = useLiveDocMeta(kind, docId);

useLiveDocSelector uses Zustand's useShallow by default so passing the same value across renders doesn't trigger updates. The renderer receives the doc + dispatch + meta from the host (see plugin author guide); inside the renderer, extension authors can use these hooks directly to narrow further.

Pending optimistic ops

When the user clicks "toggle," the renderer dispatches an op. LiveOpClient:

  1. Generates opId = crypto.randomUUID().
  2. Calls store.enqueue(kind, docId, opId, op). The reducer is applied to payload immediately so the UI shows the change instantly. The op is pushed onto pendingOps so callers can render it differently if they want ("checked, but waiting to confirm").
  3. POSTs to /api/live/:kind/:docId/op with { opId, op, baseSeq: serverSeq }.
  4. When the server's op-applied arrives via SSE with the same opId, the store removes it from pendingOps and advances serverSeq.
  5. If op-rejected arrives instead, the store rolls back: re-applies all confirmed ops from the last good state, drops the pending entry, surfaces the reason via lastError.

The reducer must be pure so the rollback replay produces identical state. This is the load-bearing reason the manifest's reducer signature accepts (state, op) with no this, no closures, no async.

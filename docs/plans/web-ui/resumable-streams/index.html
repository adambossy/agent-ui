Resumable streams

If the user's tab dies mid-response, the answer keeps streaming where it left off when they come back. The client side is the AI SDK's built-in resumeStream(); the server side is ours to implement, but we model the interface so a mock can stand in until then. Full implementation reference at /Users/adambossy/code/agent_ui/notes/resumable-streams.md.

Requirements

  1. Refreshing the tab during a streaming answer continues the answer instead of restarting it.
  2. A client developer can verify the resume flow against a local mock without spinning up Redis.
  3. The protocol marker (x-vercel-ai-ui-message-stream: v1) is set on the resume GET so the AI SDK parses it correctly.

Client side: useChat's resumeStream

The AI SDK's React hook exposes resumeStream() as a first-class action. The hook does the GET to the resume endpoint, parses the same SSE v1 events, and merges them into the same message state as a fresh stream — no duplicate code path on our side.

// src/chat/useAutoResume.ts
export function useAutoResume(chat: UseChatReturn, initialMessages: UIMessage[]) {
  useEffect(() => {
    // If the last persisted message is from the user, an assistant turn was
    // in flight when we last left this session.
    const tail = initialMessages.at(-1);
    if (tail?.role === "user") chat.resumeStream();
  }, []);
}

One effect per ChatScreen mount. The heuristic is identical to the Vercel template's — last persisted message role === "user" means the assistant's response wasn't saved yet. No extra server call to figure it out.

Backend interface (our port)

We port the shape of the resumable-stream npm package into our own library — language-agnostic, so the same conceptual API works whether the backend is Python or Node:

// Conceptual surface — same names in TS and Python.
interface StreamCache {
  // Server starts a new resumable stream.
  // Returns a tee-target the producer writes SSE bytes into; the cache
  // mirrors them to a backing store (Redis / SQLite-FTS / etc.).
  createNewResumableStream(streamId: string, source: () => AsyncIterable<Uint8Array>): Promise<void>;

  // Server replays the cached stream + tails any new bytes until end.
  resumableStream(streamId: string, makeEmpty: () => AsyncIterable<Uint8Array>): AsyncIterable<Uint8Array>;
}

// Per-session index:
interface StreamIndex {
  rememberLatest(sessionId: string, streamId: string): Promise<void>;
  getLatest(sessionId: string): Promise<string | null>;
}

Two pieces. StreamCache is the bytes; StreamIndex is the lookup ("given a session, what's the most recent streamId?"). Both have multiple backends (in-memory for tests, Redis for prod, SQLite for self-hosted dev). The HTTP layer composes them: POST creates a fresh streamId, persists it via StreamIndex, and tees through StreamCache. GET resolves the streamId and reads back.

Full reference (Vercel's original, our notes, Python port sketch): resumable-streams.md.

Mocking

The frontend's mock backend implements both POST and GET, and uses an in-memory Map<streamId, Uint8Array[]> for the cache. The POST path tees its own canned SSE script (a scripted turn we can reuse for tests) into the cache. The GET path reads it back.

For exercise: a "Refresh button" in dev mode that closes the SSE socket mid-stream lets the developer verify resume by clicking once — no actual tab refresh needed. The mock is in-memory only, so refreshing the tab still loses the cache (matching the real-world failure mode when Redis is empty / TTL'd).

Auto-resume policy

ConditionBehaviour
Opening a fresh sessionDon't resume; assistant message is fresh.
Opening an existing session, last message role === "assistant"Don't resume; the turn finished cleanly.
Opening an existing session, last message role === "user"Call resumeStream(). The GET will replay any cached bytes; if the cache is empty (TTL elapsed), the server returns an empty SSE that ends immediately and the UI shows the user message with no response — same state as if nothing was in flight.
Tapping into a subagent that's still streaming under its parentSame as above. The subagent has its own streamId; the resume endpoint replays the cached bytes for that id. If the subagent finished but its parent is still going, the resume returns the final state immediately.

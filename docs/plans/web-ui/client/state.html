State model & rendering strategy

Per-session useChat instances plus a small Zustand store for cross-cutting concerns. Pi's two-track render pattern — a stable list of completed messages plus an isolated streaming overlay — is the right approach to keep the transcript performant while tokens stream.

Requirements

  1. Streaming a long message does not slow down or repaint earlier parts of the conversation.
  2. Switching sessions is instant; the prior session's in-flight stream survives in the background and is observable when you return to it.
  3. Subagent expansions and parallel tool cards update independently — opening one does not disturb the others.

Per-session useChat

Each session id mounts its own useChat. The hook's id option is the session id; @ai-sdk/react uses it to key the internal store, so multiple sessions can be active in parallel without colliding. We hold those instances in a ChatRegistry keyed by session id; the ChatScreen component subscribes to the one matching the current URL.

// src/chat/registry.ts
const registry = new Map<string, ReturnType<typeof useChat>>();

export function useSessionChat(sessionId: string) {
  // useChat is keyed by id — multiple sessions co-exist without state collision.
  const chat = useChat({
    id: sessionId,
    transport: makeTransport(sessionId),
    onData: handleSubagentChannel,  // pulls "data-subagent-event" into the substore
  });
  registry.set(sessionId, chat);
  return chat;
}

Two-track render (pi pattern)

The transcript renders in two pieces:

  • Stable list — completed messages, keyed by message id, memo'd on identity. Never repaints when a new token arrives at the tail.
  • Streaming overlay — the single in-flight assistant message, rendered by a dedicated component that subscribes to the in-flight slice of state and re-renders on each delta.
// src/components/Transcript.tsx
export function Transcript({ chat }: { chat: UseChatReturn }) {
  const completed = chat.messages.filter(m => m.status !== "streaming");
  const inFlight  = chat.messages.find(m => m.status === "streaming");
  return (
    <Conversation>
      <ConversationContent>
        <CompletedList messages={completed} />        {/* memo by id */}
        {inFlight && <StreamingMessage message={inFlight} />}
      </ConversationContent>
      <ConversationScrollButton />
    </Conversation>
  );
}

Inspiration: pi-web-ui's MessageList + StreamingMessageContainer pair (see /Users/adambossy/code/agent_ui/pi/WEB-UI-FINDINGS.md). The pi implementation additionally deep-clones the in-flight snapshot via JSON.parse(JSON.stringify(...)) to defeat Lit's identity dirty check; we don't need that with React + Streamdown's memoisation, but the pattern is worth borrowing if profiling shows otherwise.

Subagent in-flight handling

Subagent events arrive as data-subagent-event parts on the parent's UIMessage. We do not let the AI SDK reducer treat them as ordinary message parts — instead, an onData handler intercepts the channel and forwards each event to a SubagentStore keyed by (parentToolCallId, sessionId). The SubagentTool component subscribes to that store with a selector for its own toolCallId, so each subagent expansion is an independent reactive scope.

// src/chat/subagentStore.ts (zustand)
type SubagentStore = {
  byToolCallId: Record<string, { sessionId: string; messages: UIMessage[]; status: "streaming" | "done" }>;
  apply(ev: SubagentEvent): void;
};

This keeps the AI SDK's messages array clean (it sees a single tool call, not a mountain of internal events) and gives each subagent its own update boundary.

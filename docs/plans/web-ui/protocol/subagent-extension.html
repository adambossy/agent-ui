Subagent stream extension

Subagents are modeled as tool calls. Their internal events ride on the same SSE stream as the parent, tagged with { sessionId, parentToolCallId } so the client can route them to the right inline expansion or to a tap-into target.

Requirements

  1. The user sees a subagent's reasoning, tool calls, and final answer streaming live, in-place, without leaving the parent session.
  2. Multiple subagents launched in parallel each animate independently.
  3. If the user opens a subagent as its own session, the rebuilt session looks identical to having run it standalone.

The republished model

agent_harness already supports nested subagents via republish_event_for_parent (agent_harness/core/subagents.py:272). When the parent's as_tool wrapper invokes a child, the child runs on its own EventBus, and every child event is also published to the parent's bus, bracketed by SubagentStart / SubagentStop. We adopt the same model on the wire.

Two consequences:

  • One SSE connection per top-level session. The parent's stream carries everything — its own messages, its tool calls, and any descendant subagent activity. The client never opens a second socket while a parent is live.
  • Subagent persistence is independent. Each subagent has its own session id in the harness's session store. Tapping into a subagent loads that session fresh from storage; it does not reuse the live stream.

Data shape on the wire

We wrap every republished child event in a single typed channel — data-subagent-event — with the child's session context attached. The envelope:

type SubagentEvent =
  | { kind: "start";  sessionId: string; parentToolCallId: string; agentName: string; parentSessionId: string }
  | { kind: "stop";   sessionId: string; parentToolCallId: string }
  | { kind: "event";  sessionId: string; parentToolCallId: string;
                      event: WireEvent /* a normal AI SDK v1 event */ }

// Sent as:
//   data: {"type":"data-subagent-event","data":SubagentEvent,"transient":false}

The event field carries the exact wire event the subagent would have emitted on its own stream — text-delta, reasoning-delta, tool-input-available, etc. The client's reducer demultiplexes these by (sessionId, parentToolCallId) and applies them to a nested message store, isolated from the parent transcript.

Backend note (resolved from the harness). The harness's SubagentStart dataclass (events.py:224) carries (parent_agent_name, child_agent_name, tool_call_id) but does not carry the child's session_id. However, the child Agent spawned by Agent.as_tool (agent.py:249) does have its own Session instance. The HTTP-layer adapter is responsible for attaching that session_id when it translates SubagentStart into the wire envelope above. The client must not be asked to invent it. If the adapter cannot resolve it (e.g. a subagent without a Session backend in some test path), the field is omitted and the client gracefully degrades — inline expansion still works keyed by parentToolCallId alone; only tap-into is disabled.

Worked example: two parallel subagents

// Parent invokes two subagents in one model step. The parent's stream carries:

data: {"type":"start","messageId":"m-parent"}
data: {"type":"start-step"}

// parent's reasoning, if any
data: {"type":"reasoning-start","id":"r-1"} ...

// parent issues two tool calls in parallel
data: {"type":"tool-input-available","toolCallId":"tc-A","toolName":"researcher","input":{...}}
data: {"type":"tool-input-available","toolCallId":"tc-B","toolName":"writer","input":{...}}

// each subagent starts (in any order)
data: {"type":"data-subagent-event","data":{"kind":"start","sessionId":"s-A","parentToolCallId":"tc-A","agentName":"researcher","parentSessionId":"s-root"}}
data: {"type":"data-subagent-event","data":{"kind":"start","sessionId":"s-B","parentToolCallId":"tc-B","agentName":"writer","parentSessionId":"s-root"}}

// subagent A and B emit events INTERLEAVED on the same stream — each tagged
data: {"type":"data-subagent-event","data":{"kind":"event","sessionId":"s-A","parentToolCallId":"tc-A","event":{"type":"text-delta","id":"t-A1","delta":"Looking up..."}}}
data: {"type":"data-subagent-event","data":{"kind":"event","sessionId":"s-B","parentToolCallId":"tc-B","event":{"type":"text-delta","id":"t-B1","delta":"Drafting..."}}}
data: {"type":"data-subagent-event","data":{"kind":"event","sessionId":"s-A","parentToolCallId":"tc-A","event":{"type":"tool-input-available","toolCallId":"tc-A-1","toolName":"webSearch","input":{"q":"..."}}}}
...

// each subagent stops
data: {"type":"data-subagent-event","data":{"kind":"stop","sessionId":"s-A","parentToolCallId":"tc-A"}}
data: {"type":"data-subagent-event","data":{"kind":"stop","sessionId":"s-B","parentToolCallId":"tc-B"}}

// parent receives the tool outputs (the subagents' final returns)
data: {"type":"tool-output-available","toolCallId":"tc-A","output":{"sessionId":"s-A","summary":"..."}}
data: {"type":"tool-output-available","toolCallId":"tc-B","output":{"sessionId":"s-B","draft":"..."}}

// parent continues
data: {"type":"text-delta","id":"t-parent","delta":"Based on the research..."}
...
data: {"type":"finish"}
data: [DONE]

Interleaving is fine; the client's reducer is keyed by (sessionId, parentToolCallId) and applies each event to the correct nested store.

Ordering and recursion

Three ordering rules:

  1. Within a session, events arrive in causal order. A subagent's start precedes its events, which precede its stop. The backend is responsible for enforcing this on the wire.
  2. Across sessions, ordering is undefined. Subagent A's events can interleave with subagent B's events; the client doesn't assume otherwise.
  3. Recursion depth is unbounded on the wire, bounded in the UI. A subagent of a subagent emits data-subagent-event events whose parentSessionId is the intermediate session id. The client renders one level deep inline and falls back to a tap-into link at deeper levels — see inline expansion.

A subagent's tool-output-available on the parent stream (the parent's view of the tool returning) carries the subagent's sessionId in its output field. That is the only place the tap-into target id is materialised; everything else uses the data-subagent-event envelope.

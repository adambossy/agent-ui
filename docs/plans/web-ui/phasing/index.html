Phasing: MVP → iterations

Ship a minimum-viable chat surface against the mock backend in week 1, then add subagent inline expansion, then resume, then file uploads, then the session tree. Each phase ends with a runnable app.

Requirements

  1. Each phase produces a runnable, useful app on its own — no phase requires the next to be valuable.
  2. The frontend can be developed end-to-end without a real backend by relying on mocks defined in earlier branches.
  3. Every phase ships responsive — verified on a 375px-wide viewport before that phase is called done.

MVP (Phase 1)

The smallest thing that talks to the AI SDK protocol and shows reasoning + tool calls + assistant text.

  • Vite + React + Tailwind + shadcn + AI Elements installed (Conversation, Message, Reasoning, Tool, PromptInput).
  • One route: /c/:sessionId?. useChat wired to DefaultChatTransport.
  • Mock backend: a single canned turn covering reasoning, one tool call (with custom WeatherRenderer), and assistant text. Scripted SSE replay.
  • Two-track render skeleton (stable list + streaming overlay), even though MVP only has one turn.
  • No subagents, no uploads, no resume, no session tree, no auth.

Exit criteria: open the app, see the canned turn stream in, reload, see it persist — and the same flow works on a 375px-wide viewport (phone breakpoint) with no layout breakage. Approximate effort: 3–5 days.

Phase 2: subagents

  • Add data-subagent-event handling — onData hook splits child events into a Zustand SubagentStore.
  • Implement SubagentTool + SubagentTranscript (one level deep; deeper nests render as tap-into headers only).
  • Add openSession(sessionId) primitive + the /c/:sessionId hydration path (calls getSession()).
  • Mock backend: extend the canned turn to fire 2 parallel subagents, each with its own reasoning + tool calls.
  • Visual: agent-color tinting, status badges, "Open ↗" tap-into.

Exit criteria: drive the canned subagent turn, expand both inline, tap into one, navigate back, expand a different one. Approximate effort: 5–7 days.

Phase 3: resume + uploads

  • Wire resumeStream() through useAutoResume. Mock backend simulates a disconnect mid-stream.
  • Implement UploadAdapter interface + MockUploadAdapter + composer chip UI + file-part attachment on send.
  • "Refresh in the middle" dev affordance to verify resume by click rather than browser refresh.

Exit criteria: start a canned turn, click "kill stream" in dev mode, refresh, see the response continue. Drag a file in, send it, see it as a chip in the resulting user message. Approximate effort: 4–5 days.

Later phases

  • Phase 4 — Session tree. Sidebar tree view, GET /api/sessions/tree wiring, live updates as new sessions appear.
  • Phase 5 — Real backend. Swap the mock for the real agent_harness HTTP layer. Most of this is server-side; on the client it's an env var flip.
  • Phase 6 — Tool approvals. Wire addToolApprovalResponse. Surface approval prompts inside SubagentTool headers and inside ordinary tool cards.
  • Phase 7 — Generative UI polish. Build out the rest of the per-tool renderers (search results, diffs, tables). Theme system extensions.
  • Phase 8 — Production concerns. Auth, telemetry, error reporting, SSR for the marketing surface, accessibility audit.

Phases 4 onward are negotiable in order; each can stand alone once Phase 3 is done.

Tap-into: opening a subagent as a first-class session

A single primitive — openSession(sessionId) — handles both this and session-tree navigation. The URL becomes /c/<sessionId>, the chat surface rehydrates from GET /api/sessions/:id, and the session becomes the user's primary context until they navigate away.

Requirements

  1. Tapping a subagent header puts the user inside that subagent's session as if they'd opened it directly — same URL, same history, same back button.
  2. If the subagent is still streaming live in its parent, opening it as a first-class session attaches to the live stream via resume; the user sees tokens continue to arrive.
  3. The parent's tab does not lose state when the user navigates away — coming back is instant.

The primitive

// src/navigation.ts
import { useNavigate } from "react-router-dom";

export function useOpenSession() {
  const navigate = useNavigate();
  return (sessionId: string, opts?: { newTab?: boolean }) => {
    if (opts?.newTab) {
      window.open(`/c/${sessionId}`, "_blank");
      return;
    }
    navigate(`/c/${sessionId}`);
  };
}

That's it. Tap-into of a subagent, session-tree click, and "open in new tab" all funnel through this. The receiving route (/c/:sessionId) handles hydration, resume, and rendering — see client skeleton.

URL & history

  • URL is the source of truth. Sessions are addressable by id; the client never tracks "which session am I on" in component state separately.
  • Browser back/forward works naturally. Navigating into a subagent then hitting Back returns to the parent at the scroll position it was when you left. The ChatRegistry keeps the parent's useChat alive in the background; the route swap merely changes which one the surface is bound to.
  • Deep-linking on first load is supported. /c/<sessionId> works whether you got there from a click, a refresh, or a paste into the address bar. The hydrating GET is the only path; there is no client-side cache that would shortcut it.
  • Optional query param ?from=<parentSessionId> lets the breadcrumb show "← back to parent" without a server round trip. Purely UX sugar; the URL still resolves without it.

What the tap-into target renders

Once at /c/<sessionId>:

  1. Hydrate. GET /api/sessions/:sessionId returns { session, messages: UIMessage[] }. The session metadata includes parentSessionId and parentMessageId when applicable.
  2. Show the breadcrumb. If parentSessionId is set, a top bar reads: 
    ← Parent <agent-name>  /  <this agent name>
    Clicking "Parent" calls openSession(parentSessionId). Same primitive.
  3. Mount useChat for this session. If session.status === "streaming", fire resumeStream() immediately (the AI SDK hook handles the GET to the resume endpoint).
  4. Render the transcript. Exactly the same transcript component as the root — including the ability to inline-expand any subagents this session itself launched.

The deepest user-visible nesting therefore is always: a top-level chat surface plus one level of inline subagent expansion. Going deeper means tapping in, which makes the deeper subagent the new top-level — a tree walk one step at a time, never a wall of nested DOM.

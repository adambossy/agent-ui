Subagents UX

A subagent invocation looks like a tool call with a fancy renderer. It can be expanded inline one level deep to show the subagent's full transcript (reasoning, tool calls, assistant text). Tapping the header opens the subagent as a first-class session. Parallel subagents each animate independently.

Requirements

  1. A user can tell at a glance which subagent is running, which has finished, and which failed — even with several running in parallel.
  2. A user can expand a subagent inline and read its full activity without losing context in the parent.
  3. A user can tap a subagent header to open it as a top-level session and read or continue it there.

Model: subagents are tool calls

From the client's perspective there is no separate "subagent" concept on the parent's UIMessage. The parent has a regular tool-<subagent-name> part whose input is the subagent's prompt and whose output is the subagent's final return value plus its sessionId. Everything that makes it special is delivered as data-subagent-event records on the same stream — see protocol extension.

The ToolRendererRegistry picks the SubagentTool component over the default Tool when the tool's metadata flags it as a subagent (a registry flag set when the tool is registered, or — fallback — a name match against a known subagent registry). Everything else routes to the default Tool or a custom renderer.

Lifecycle states

The SubagentTool derives its visible state from the tool-call state machine plus the substore:

State (tool-call)What the user sees
input-streaming / input-available"Starting <agent-name>…" — disabled chevron, no expansion yet.
received data-subagent-event kind=start"<agent-name> · running" — chevron live, expansion shows live transcript.
received data-subagent-event kind=stop"<agent-name> · finished in N s" — expansion now read-only.
output-availableHeader gets a tap-into link "Open as session" (uses the sessionId in the output).
output-errorHeader turns red; expansion still readable; "Open as session" still works.

Header row anatomy

┌─────────────────────────────────────────────────────────────────────┐
│  ▸  ✦ researcher    ● running   12 toolcalls · 8s    Open ↗        │
└─────────────────────────────────────────────────────────────────────┘

Left to right: chevron (expand/collapse), an agent-tinted glyph, the agent name, a status badge (with pulsing dot while running), a live counter of tool calls + elapsed time, and a tap-into link. Clicking the chevron toggles the inline expansion; clicking the agent name or "Open ↗" calls openSession(sessionId).

Visual tint comes from the agent name (hashed to a stable hue, like opencode's agent colors). Parallel subagents thus get distinct colors, making it easy to track them on a busy turn.

Permissions and approvals (later)

agent_harness supports NestedInterruption — when a subagent has pending approvals, the parent's ToolDispatch rolls them into the parent's approval queue. The UI surface for this is the AI SDK's tool-approval states (approval-requested, approval-responded), exposed via useChat's addToolApprovalResponse — same primitive the Vercel template uses for the weather Allow/Deny prompt.

Where the approval prompt appears: always in the parent's composer area — a banner directly above the input — never inside a subagent expansion. Rationale: the composer is the user's stable focus point; bouncing approvals between nested expansions is easy to miss. The banner names the agent that needs the approval (e.g. "researcher needs to run webSearch: Allow / Deny") and the corresponding subagent expansion can pulse / highlight to surface where the request originated.

Out of scope for MVP; documented here so the layout reserves room for the composer-area banner from day one.

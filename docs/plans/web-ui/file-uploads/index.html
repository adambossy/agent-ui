File uploads

Drag-drop a file, watch it upload, see it as an attachment chip on the next message. The frontend is the focus; the backend just has to satisfy a 4-method interface. A mock implementation simulates a slow upload locally so the entire flow is exercisable without a real bucket.

Requirements

  1. A user can drag a file onto the composer (or click an attach button) and see upload progress in-line.
  2. An upload can be canceled before it finishes; a failed upload can be retried without losing the file.
  3. Once uploaded, the file appears as a chip on the composer; sending the message attaches it as a file part on the outgoing UIMessage.

UX flow

  1. User drags a file (or hits the paperclip icon and picks one).
  2. A chip appears on the composer in "uploading" state with a thin progress bar.
  3. On success, the chip shows a thumbnail (images), a file icon (everything else), the filename, and the size. An × button removes it.
  4. On error, the chip turns red with a "Retry" button.
  5. When the user submits the message, every "ready" chip is attached as a { type: "file", url, filename, mediaType } part on the UIMessage. "Uploading" chips block the send button.

One chip per file. AI Elements' Attachments component renders the post-send view inside the message bubble.

The interface (what we need from the backend)

// src/uploads/types.ts
export interface UploadAdapter {
  /** Upload a single file. Returns when the file is durably stored. */
  upload(
    file: File,
    opts: {
      signal: AbortSignal;
      onProgress?: (loaded: number, total: number) => void;
    }
  ): Promise<UploadResult>;

  /** Optional: pre-signed URL flow. If supported, the client uses this path
   *  to skip server bandwidth. */
  getPresignedPut?(args: { filename: string; mediaType: string; size: number })
    : Promise<{ uploadUrl: string; finalUrl: string }>;
}

export type UploadResult = {
  url: string;          // stable, server-relative or absolute
  filename: string;
  mediaType: string;
  size: number;
};

The backend's POST /api/uploads implements upload(). Whatever it does internally — S3 multipart, R2 single PUT, local disk — is opaque to us. If the backend supports presigned PUT URLs, the adapter implements getPresignedPut and the client switches to direct-to-bucket on its own (no extra wiring).

Mocking

// src/uploads/mock.ts
export class MockUploadAdapter implements UploadAdapter {
  async upload(file, { signal, onProgress }) {
    // Simulate 2 seconds of progress in 100ms ticks.
    const ticks = 20;
    for (let i = 1; i <= ticks; i++) {
      if (signal.aborted) throw new DOMException("aborted", "AbortError");
      await new Promise(r => setTimeout(r, 100));
      onProgress?.(file.size * i / ticks, file.size);
    }
    // Optionally simulate a 10% failure rate for development:
    if (Math.random() < 0.1) throw new Error("mock upload failed");

    // Echo the file back as a blob URL so the chat can preview it locally.
    const url = URL.createObjectURL(file);
    return { url, filename: file.name, mediaType: file.type, size: file.size };
  }
}

The mock exercises every state the real adapter would — progress, success, abort, retry — so the UI is fully wired before a backend exists. Switching to the real adapter is one factory call.

Client state for in-flight uploads

// src/uploads/store.ts
type UploadEntry = {
  tempId: string;            // client-side stable id
  file: File;
  status: "uploading" | "ready" | "error" | "canceled";
  progress: number;          // 0..1
  result?: UploadResult;
  error?: string;
  controller: AbortController;
};

export const useUploadStore = create<{
  entries: Record<string, UploadEntry>;
  enqueue(file: File): string;
  cancel(tempId: string): void;
  retry(tempId: string): void;
  remove(tempId: string): void;
  attachReadyTo(message: ComposerDraft): ComposerDraft;
}>(/* ... */);

The composer subscribes to "ready" entries and renders chips. On submit, attachReadyTo turns those into file parts on the outgoing UIMessage and clears them from the store.

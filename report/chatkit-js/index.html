OpenAI ChatKit

ChatKit is not actually open source where it counts. The three public repos (Apache-2.0 / MIT) ship only TypeScript declarations + a React forwarder. The actual chat UI is a closed minified bundle served from cdn.platform.openai.com inside an iframe, hard-bound to OpenAI's domain allowlist, Responses API, and Agents SDK. It is the same renderer that powers ChatGPT.com.

Headline answers.
  • Open source? No — the runtime is not in the public repos. Only types + a wrapper.
  • Self-host outside OpenAI? No, not as shipped. Four hard couplings (CDN, domain allowlist, managed-session API, Responses-API types).

4. Architecture

Component model. One web component: <openai-chatkit>. The React layer (packages/chatkit-react/src/ChatKit.tsx:38-107) renders that custom element, calls el.setOptions(control.options) after customElements.whenDefined('openai-chatkit') resolves, and wires every event through EVENT_HANDLER_MAP. useChatKit (useChatKit.ts:13-22, 58-108) exposes 8 imperative methods: focusComposer, setThreadId, sendUserMessage, setComposerValue, fetchUpdates, sendCustomAction, showHistory, hideHistory.

Thread-item types (extracted from the bundle):

  • assistant_message, user_message — text turns.
  • client_tool_call — UI-side tool invocation card.
  • end_of_turn — turn boundary.
  • generated_image, image_generation — progressive image generation.
  • structured_input — form-style input.
  • task — reasoning / "thinking" summary.
  • widget — server-authored UI fragment (see §7).
  • workflow — multi-step plan.

5. Streaming protocol

SSE over a single HTTPS endpoint (POST /chatkit; starter/chatkit/backend/app/main.py:25-35; returns text/event-stream when result is StreamingResult). Not Chat Completions delta and not raw Responses API — it's ChatKit's own thread-event protocol:

  • ThreadItemDoneEvent, ThreadItemReplacedEvent, ThreadItemUpdated
  • ProgressUpdateEvent
  • ClientEffectEvent
  • WidgetRootUpdated

However, it is layered on the Responses API: chatkit.agents.stream_agent_response converts openai-agents Runner output (which emits Responses API events) into ChatKit events (advanced/examples/cat-lounge/backend/app/server.py:110-113). The Python chatkit.types re-exports openai.types.responses.ResponseInputContentParam, ResponseInputImageParam, ResponseInputTextParam directly.

Rendering happens inside the closed iframe, so the visual treatment of reasoning, tool calls, etc. can only be described from the protocol surface — not from source. The iframe's React SPA is minified and unowned by the user.

7. Widget DSL — the distinctive piece

ChatKit's signature feature: a server-authored, client-rendered widget protocol. Roughly 25 components in types/widgets.d.ts:1-598:

GroupComponents
ContainersCard, Box, Row, Col, Form
TextMarkdown, Title, Caption, Text, Label, Badge
MediaIcon, Image
InputsInput, Textarea, Select, DatePicker, RadioGroup, Checkbox, Button
ListsListView, ListViewItem
TableTable, Table.Row, Table.Cell
LayoutDivider, Spacer, Transition

Widgets are JSON authored on the server (e.g. cat_agent.py builds a Card with a ListView of cats) and rendered by the closed iframe.

The catch. Only ChatKit's renderer can render ChatKit widgets. The widget DSL is rich, but it's locked to OpenAI's UI.

8. Auth and provider switching

ModeConfigAuth flow
Self-hostedCustomApiConfig = { url, domainKey, fetch?, uploadStrategy? }api.url can be any URL; cookie/header auth between browser and your /chatkit endpoint is your problem (inject via custom fetch). domain_pk_localhost_dev is the universal local placeholder; production requires registering your domain at platform.openai.com.
ManagedHostedApiConfig = { getClientSecret }Your server calls api.openai.com/v1/chatkit/sessions with OPENAI_API_KEY + workflow id to mint a client_secret; the iframe authenticates each request with that secret.

There is no ChatGPT user token concept — auth is between your app and OpenAI's API + iframe; ChatGPT.com login is not involved. But the domain_pk_... is permanently tied to a platform.openai.com organization.

Can the OpenAI key be swapped for Anthropic / Azure / OpenRouter? — in theory; practically, no.

Not without writing a custom ChatKitServer.respond() from scratch:

  • Every example targets openai-agents + gpt-4.1-mini.
  • chatkit.types is implemented in terms of openai.types.responses.*.
  • grep for Anthropic / claude / azure / OpenRouter / gemini across all three repos returns zero hits.

The SSE protocol is in principle provider-agnostic, so a determined developer could yield ChatKit events from any source — but you would re-implement the converter, lose image-gen partial streaming, lose reasoning-summary support, lose the Agent Builder workflow runtime — and still hit OpenAI's CDN + domain allowlist.

9. Same renderer as ChatGPT.com

The bundle's capability profiles literally name three modes — visible in chatkit.js:

Xe = {
  chatkit:                { allow: [...] },
  "chatgpt-shell":        { ... },
  "chatgpt-shell-anonymous": { ... }
}

ChatGPT.com appears to be a ChatKit consumer where the chatgpt-shell profile enables additional commands (command.shareThread, event.thread.restore, event.message.share, event.image.download, command.setTrainingOptOut).

Implication. For end users this means ChatKit integrations will feel exactly like ChatGPT, because they are the same renderer. For developers it means you are renting a UI OpenAI controls end-to-end — they ship updates to your users without you redeploying, but you cannot patch a bug, customize a render path, audit DOM behavior, fork to add a feature, host a stable older version, or run an air-gapped deployment.

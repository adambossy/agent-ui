Session tree & navigation

A sidebar tree that mirrors the harness's session graph: every session, with parents and children, navigable. Clicking a node opens it via the same openSession() primitive used for subagent tap-into. The pi web UI's session list is the visual reference; the structure is opencode's parent/child sessions.

Requirements

  1. A user can see every session they've started and every subagent invoked under them, with one-click navigation to any node.
  2. The tree updates as new sessions are created — switching to a new conversation should show up without a full page reload.
  3. The tree's shape is whatever the backend gives us; the client does not assume a particular depth or fan-out.

Two stacked views: Recent (flat list of last N top-level sessions; persists in localStorage) and Tree (the full graph fetched from the server). The current session is highlighted in both. Subagent sessions appear inside the tree under their parent; the parent's tree item is expandable. The number in parentheses next to a subagent indicates how many children it has of its own — a single nudge that deeper subagent activity is available without forcing the user to scroll a giant tree.

Fetching the tree

// returned by GET /api/sessions/tree
type SessionTreeNode = {
  sessionId: string;
  title: string;                  // server-generated; may be empty for unnamed sessions
  agentName: string;              // "user" for root sessions; agent name for subagents
  status: "idle" | "streaming" | "error";
  parentSessionId?: string;
  parentMessageId?: string;
  createdAt: string;              // ISO
  children: SessionTreeNode[];    // recursive
};

Loading strategy: fetch once on app boot, plus on every URL change (stale-while-revalidate via SWR or Tanstack Query). For very large trees, the contract permits a ?rootId=... query and ?depth=N to fetch a subtree — but the MVP fetches the full tree and assumes it fits.

Live updates: when the client receives data-subagent-event with kind: "start" on any session's stream, it optimistically inserts a placeholder node into the tree (status: streaming) under the parent. On kind: "stop", it triggers a tree refetch to pick up the real node + title.

The "open new session" primitive

One client-side function — openSession(sessionId) from tap-into — drives:

  • Tapping a subagent header in the chat.
  • Clicking a tree node in the sidebar.
  • Clicking a "← Parent" breadcrumb on a subagent session.
  • Following a deep link on first load (the router invokes the same code path).

It is a thin wrapper around react-router's navigate — the heavy lifting (hydration, resume, store mounting) happens in the route effect. Keeping it one function means we never accidentally have two ways to enter a session that diverge.

"New session" (the + button) is a different primitive: createSession() hits POST /api/sessions, gets a fresh id, then calls openSession(newId). The "create a subagent session" path is the same — but the caller passes { parentSessionId, parentMessageId } so the backend records the lineage.

Deep linking & URL shape

URLBehaviour
/New session, blank composer.
/c/<sessionId>Opens that session. Resumes if streaming.
/c/<sessionId>?from=<parentId>Same, plus shows a "← back to parent" affordance.
/tree (modal route)Opens the session-tree drawer on top of the current session. Closing returns to where you were.

Tree node ids are opaque strings; the client never tries to parse them. The server is free to use UUIDs, ULIDs, slugs, or whatever else makes sense.

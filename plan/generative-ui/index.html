Generative UI registry

Tool calls can render whatever the tool's renderer says they should. The Vercel template's <Weather> card is the canonical example; we keep that pattern and formalise it as a per-tool registry with a default JSON fallback.

Requirements

  1. A frontend developer can add a new tool renderer with one file plus one registration line, no changes to chat-engine code.
  2. Tools without a custom renderer still render usefully (collapsible JSON args + output, status badge).
  3. A renderer can react to streamed args (e.g. show a partial chart as numbers arrive) without race conditions.

The registry

// src/tools/registry.ts
type ToolRenderer = React.ComponentType<{ part: ToolUIPart }>;

const renderers = new Map<string, ToolRenderer>();
const subagentNames = new Set<string>();          // tool names known to be subagents

export function registerToolRenderer(name: string, R: ToolRenderer) {
  renderers.set(name, R);
}
export function markAsSubagent(name: string) {
  subagentNames.add(name);
}

export function resolveToolRenderer(part: ToolUIPart): ToolRenderer {
  // tool-getWeather → "getWeather"
  const name = part.type.slice("tool-".length);
  if (subagentNames.has(name)) return SubagentTool;     // see Subagents UX
  return renderers.get(name) ?? DefaultTool;
}

Registration happens in src/tools/register.ts at module load — colocated with the renderer files. No central switch statement; adding a renderer is an additive edit.

Renderer contract

A renderer is a React component that takes { part: ToolUIPart }. It receives the part on every state transition:

part.stateWhat's availableTypical render
input-streamingpartial input (may be undefined)skeleton or placeholder
input-availablecomplete inputloading state ("Fetching weather for …")
output-availableinput + outputfull custom render
output-errorinput + errorTexterror card
// src/tools/renderers/Weather.tsx
export function WeatherRenderer({ part }: { part: ToolUIPart }) {
  if (part.state === "input-available")
    return <LoadingCard>Fetching weather for {part.input.location}…</LoadingCard>;
  if (part.state === "output-available")
    return <WeatherCard data={part.output} />;       // the gradient card
  if (part.state === "output-error")
    return <ErrorCard message={part.errorText} />;
  return null;
}
registerToolRenderer("getWeather", WeatherRenderer);

Default fallback

DefaultTool is AI Elements' <Tool> + <ToolHeader> + <ToolInput> + <ToolOutput> — a collapsible card with a status badge, the JSON-stringified input under "Parameters", and the output under "Result". Strings render as plain text; objects as a JSON code block; React elements render directly (in case the tool itself returns JSX through some encoding). Nothing fancy, but it's always correct.

Most tools we ship will use the default. Custom renderers are reserved for things where a structured render genuinely helps the user (weather card, file diff, map, table).

Examples to ship

  • getWeather — keep the gradient card from the Vercel template (we already verified it during the live walkthrough).
  • webSearch — list of result snippets with favicons; partial render as results arrive.
  • readFile — collapsed file preview with line numbers; expand to view full.
  • editFile — unified diff via a CodeMirror-based viewer; reuse AI Elements' Artifact-internal diff rendering only if it's easy to lift, otherwise roll our own with diff + react-diff-viewer-continued.
  • todoWrite — checklist (matches opencode's TodoWrite block).
  • Subagent tools — flagged via markAsSubagent(name); routed to SubagentTool instead of the default.

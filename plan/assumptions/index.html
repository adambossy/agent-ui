Resolved (read from ~/code/agent_harness) + remaining open questions

Resolved

  1. SubagentStart does not carry a child session_id today. SubagentStart(parent_agent_name, child_agent_name, tool_call_id) (events.py:224). However, the child Agent does have its own Session instance with its own session_id — see Agent.as_tool (agent.py:249) which calls agent.run(prompt, …, event_bus=child_bus). The HTTP-layer adapter is responsible for threading the child session_id into the wire event when it republishes; the plan documents this requirement on subagent extension and treats it as the smallest backend ask.
  2. The Session Protocol does not track parent/child linkage. Surface is just session_id, get_messages, add_messages, clear, plus run-state methods (memory.py). Implication: the HTTP layer maintains its own (child_session_id → parent_session_id, parent_tool_call_id) index. The session-tree endpoint reads from there.
  3. Subagents share the parent's stream; they are not independently resumable today. RepublishingBus forwards every child event onto the parent's bus (subagents.py:272). One SSE stream per top-level run. The plan adapts: tap-into of an in-flight subagent attaches to the parent's resume stream and filters by tool_call_id (see tap-into); tap-into of a completed subagent rehydrates from the child's persisted Session via GET /api/sessions/:childId.
  4. Auth. Punted. The client will work without an auth layer in local dev; whoever stands up the production HTTP layer chooses (cookie or bearer) and the client transport is configured accordingly.
  5. Nested-subagent approval UX. Approval prompts always surface in the parent's composer, not inside the subagent's header or transcript. Rationale: the parent's composer is the user's stable focus point; bouncing approvals between nested expansions makes them easy to miss.
  6. Per-session feature flags. No. The protocol does not carry them; the harness does not model them; we don't speculate.
  7. Responsive design defaults. Accepted as drafted — sidebar collapses to a drawer below md, composer sticky-bottom with safe-area insets, 44 × 44 px touch targets, inline subagent expansion shrinks to ~12 px indent on phone with the existing "tap-into only" fallback documented as an active escape hatch (verified in Phase 2).

Remaining open

  1. The exact HTTP-layer adapter for translating SubagentStart + republished child events into data-subagent-event with session_id attached. The harness has the information (the child's RunContext has the Session); the adapter needs ~20 LOC to thread it through. Out of scope for the frontend plan but flagged here so the backend implementer doesn't miss it.
  2. How long does the backend hold a child session's stream-attached state after the parent finishes? For "tap into a still-streaming subagent while parent is also still streaming," the child's events ride the parent stream. After the parent finishes, the child's already done. So this is mainly relevant if the parent's stream-cache TTL is shorter than the user's typical "open it later" window — informs Redis TTL choices in resumable streams.
  3. Mobile testing target devices. The plan commits to a 375 px viewport (iPhone SE / mini). Whether to add automated mobile-Safari testing (BrowserStack / Playwright mobile emulation) is a Phase 8 call.

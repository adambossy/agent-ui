Agent web UI on the AI SDK protocol, against agent_harness

A Vite + React app that mimics the Vercel chatbot template's chat surface, drops the artifact side-panel, keeps generative tool rendering and file uploads, and adds two things the template lacks: subagents-as-tool-calls with one-level inline expansion + tap-into, and a navigable session tree. The backend is agent_harness; this plan defines only what the client expects from it.

Requirements

  1. A user can send a message, see reasoning and tool calls stream in, and read the answer with the same polish as ChatGPT or the Vercel chatbot demo.
  2. When the agent delegates to a subagent, the user sees the subagent's full activity inline (one level deep) and can open the subagent as its own first-class session.
  3. The user can navigate the full tree of sessions in a sidebar and open any of them.
  4. If the user refreshes the tab mid-stream, the answer keeps streaming where it left off.
  5. The same client works against the real backend and against a mock used in development, with no code changes.
  6. The same app is fully usable on a phone, a tablet, and a laptop/desktop — every screen, every action, every state — without a separate mobile build or a feature-stripped variant.

The goal

Build a web UI for agent_harness that feels like the Vercel chatbot template, minus the artifact side-panel, plus the subagent and session-tree affordances of sst/opencode and pi. Reuse the Vercel AI SDK client (@ai-sdk/react's useChat + AI Elements) but speak the protocol against a Python agent_harness backend. Define a thin, language-agnostic protocol so the same client can target a mock during development.

Scope

In: the client app; the protocol contract; the client-side libraries we'll port from resumable-stream; the interfaces we expect the backend to satisfy; mocks for the backend pieces so the frontend can be developed end-to-end without a live agent_harness instance.

Out: any backend implementation. agent_harness already has its own event taxonomy (RunStart / MessageDelta / ToolCallStart / SubagentStart / etc.) and its own session backends (InMemory / Sqlite / Redis). This plan describes the seam only — what the harness has to expose over HTTP for the client to do its job. Gaps in the harness today are listed as open questions, not as work items.

Strategy

  1. Two protocols, one client. The wire format is the Vercel AI SDK UI Message Stream v1 — text-*, reasoning-*, tool-input-*, tool-output-*, data-*. The harness's native Event taxonomy is mapped to it on the way out of the HTTP layer (server-side, not by us).
  2. Subagents-as-tool-calls. The harness already exposes subagents via build_agent_tool + republish_event_for_parent; we mirror that on the wire as a regular tool-<subagent-name> part whose output carries the child session id, plus inline data-subagent-event records that the client renders into a nested mini-transcript.
  3. "Open new session" is one primitive. A single client-side function — openSession(sessionId) — drives both tap-into-a-subagent and session-tree navigation. The URL is the source of truth (/c/<sessionId>).
  4. Two-track rendering, pi-style. Stable list of completed messages + isolated streaming overlay. Avoids the entire-transcript-repaints-on-token problem.
  5. Mock first. Every backend boundary (file upload, resume, session tree fetch, chat POST) ships behind an interface with a mock impl. The MVP can be built end-to-end before agent_harness has its HTTP layer.
  6. Mobile is a first-class target. The chat surface, the composer, the sidebar tree, subagent expansions, file uploads, and the model picker all have to work on a 375px-wide viewport. Where the desktop affordance does not translate — most notably the left sidebar — the mobile layout substitutes a drawer or a sheet rather than dropping the feature.

The tradeoffs we're making explicit

  • Inline subagent events vs separate streams. The harness republishes child events onto the parent's bus. We adopt that on the wire. Pro: one SSE connection per top-level session, the parent UI gets live subagent activity for free. Con: a subagent that's tapped-into is rebuilt from persistence, not resumed from a live socket. That's the deal opencode strikes too.
  • No artifact side-panel. We drop the 60%-wide editor entirely. The space is reclaimed for inline subagent expansions. Generative UI replaces it — custom tool renderers can still produce rich panels inside the chat column.
  • AI SDK v1 protocol, not a custom one. Buys us useChat, DefaultChatTransport, AI Elements components, the entire React ecosystem. Costs us a translation layer on the backend; agent_harness emits its own events and someone has to convert them to AI SDK shapes.
  • One level deep, then navigate. Inline expansion of a subagent shows its full activity but does not recursively inline its own subagents. Going deeper is a tap-into. Matches the opencode pattern and keeps the DOM bounded.

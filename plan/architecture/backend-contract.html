Backend contract

What the HTTP layer in front of agent_harness must expose for our client to do its job. Defines endpoints, request/response shapes, and the touchpoints to agent_harness Protocols. Backend implementation itself is out of scope — this is the seam.

Requirements

  1. A frontend developer can wire the client to a real backend or a mock without code changes — same endpoint shapes either way.
  2. Anyone reading this can tell, in five minutes, what the backend has to deliver before the client is useful.

Session store expectations

The harness's Session Protocol already stores messages and run-state snapshots per session id. For our purposes we additionally need:

  • Parent linkage. Each session has an optional parentSessionId and parentMessageId (the message in the parent that invoked this subagent). The harness's SubagentStart event already carries the child agent name; we need the child session id persisted alongside.
  • Tree query. Some way to walk from a root session to all descendants. The client doesn't care if it's a recursive CTE, a materialised path, or a graph — just that GET /api/sessions/tree returns the shape it expects.
  • Message persistence in the AI SDK shape. The harness stores its own Message objects; the backend HTTP layer is responsible for converting them to UIMessage (with parts: UIMessagePart[]) on read. See protocol mapping.

Auth

Whatever the backend's choice is, the client just needs the same auth-bearing identity on every request — cookie, bearer token, or header set by a proxy. The AI SDK's DefaultChatTransport sends credentials: 'include' by default for cookies; or we configure the transport with a fetch wrapper that adds an Authorization header. No further auth ceremony is part of the protocol; the SSE events do not carry user identity.

  resumeStream(sessionId: string): ReadableStream<Uint8Array>;

}
